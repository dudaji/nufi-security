# KR_LOCATION 오차 분석 — false-negative 덤프·분류 (v0.2.0 주소 정확도 착수 단계)

목적: 골드셋의 `KR_LOCATION` false-negative(놓친 케이스)를 **전량 덤프·분류**해,
후속 대응단계(P2 규칙확장 / P3 유니온 / P4 조건부 미세조정)의 우선순위를 데이터로
확정한다. 이 단계는 코드·규칙을 바꾸지 않는 **측정·분류 산출물**이다.

- 재현(에어갭·결정적): `python3 scripts/dump_kr_location_fn.py`
- 원자료(JSON): `docs/reports/kr-location-fn-dump.json` (행 단위 FN·경계 케이스 전량)
- 프로덕션 기준선: `docs/reports/baseline-int8.json` (onnx-int8, test split)
- 골드셋: `samples/gold/{test,dev}.jsonl` (전량 합성, CC0). `KR_LOCATION` 40건(test 24 / dev 16)

## 1. 백엔드별 KR_LOCATION recall (전 40건)

| 백엔드 | split | 분모 | 적중 | FN | recall |
|---|---|---:|---:|---:|---|
| onnx-int8 (프로덕션) | test | 24 | 19 | **5** | 0.7917 (CI95 0.595–0.908) |
| gazetteer (폴백/에어갭) | test | 24 | 15 | 9 | 0.625 |
| gazetteer (폴백/에어갭) | dev | 16 | 8 | 8 | 0.500 |

- 프로덕션 목표선은 0.90+. 현재 프로덕션 recall 0.79 → **갭 = 5건(test)**.
- 에어갭 환경에는 프로덕션 모델(onnx-int8)이 미프로비저닝이라 이 리포트의 **행 단위 덤프는
  gazetteer 백엔드로 결정적 재현**한다. 프로덕션 FN(5건)은 gazetteer FN(9건, 아래 분류)의
  **부분집합**이며 — 두 백엔드 모두 오차가 동일 클래스(어휘밖 고유지명)에 집중된다.
  프로덕션 5건의 정확한 행 지목은 모델 프로비저닝 후 `--backend onnx-int8` 재실행으로 확정 가능.

## 2. false-negative 클래스 분류

분류 스키마: (a) 도로명주소(로/길+번지) · (b) 상세주소(동/호·건물명) · (c) 어휘밖
고유지명(신도시·개발지구·랜드마크·복합몰) · (d) 경계/부분매치 · (e) 기타.

### 2.1 클래스 미탐지 FN (gazetteer, test+dev 40건 기준) — 전량 class (c)

| 클래스 | 하위 유형 | 건수 | 비중 | 예시(골드 지명) | 권장 대응 |
|---|---|---:|---:|---|---|
| (c) 어휘밖 고유지명 | 복합몰/상업 | 5 | 29% | 코엑스몰·센텀시티 | P2 → P3 |
| (c) 어휘밖 고유지명 | 산업/테크노단지 | 4 | 24% | 판교테크노밸리 | P2 |
| (c) 어휘밖 고유지명 | 신도시/국제도시 | 3 | 18% | 송도국제도시 | P2 |
| (c) 어휘밖 고유지명 | 공원/랜드마크 | 3 | 18% | 광교호수공원·동대문디자인플라자 | P2 |
| (c) 어휘밖 고유지명 | 개발지구 | 2 | 12% | 마곡지구 | P2 |
| **합계** | | **17** | **100%** | 7개 고유명(중복 포함 17행) | |

- **class (a) 도로명주소 = 0건, class (b) 상세주소 = 0건, class (e) 기타 = 0건.**
- 놓친 FN 은 예외 없이 `gazetteer_unlisted=True` 로 태깅된 **어휘밖 고유지명**이다.
  행정구역(부산광역시·성남시 분당구 등 `unlisted=False` 12건)은 접미사 규칙으로 전량 탐지된다.

### 2.2 class (d) 경계/부분매치 (gazetteer 14행)

접미사 규칙이 **다토큰 행정 스팬을 조각내거나**(예: `성남시 분당구`→`성남시`+`분당구`),
알려진 지명과 접미사 규칙이 **중복 스팬을 배출**해 골드 경계와 정확히 일치하지 않는 경우다.
대표 예: `성남시 분당구`·`강남구 역삼동`·`용인시 기흥구`(다토큰 행정) · `여의도 IFC`(라틴 혼용).

> 주의: (d) 는 gazetteer 특유의 산출물(중복·조각 스팬)이며 프로덕션 span-exact 품질과
> 동일하지 않다. 클래스 recall(행 단위)에는 hit 으로 잡히지만 스팬 경계 품질(P3 유니온의
> 실제 대상)을 대변한다. onnx-int8 의 경계 품질은 모델 프로비저닝 재실행으로 별도 측정 권장.

## 3. 결론 — 후속 P2/P3/P4 우선순위 권고

프로덕션 KR_LOCATION FN(0.79→0.90+ 갭 5건)은 **100% class (c) 어휘밖 고유지명**이며,
모두 규칙적 접미사(신도시/국제도시·지구·테크노밸리/밸리·몰/플라자/센터/시티·공원/광장)를
갖는다. 따라서 **P2(규칙확장)를 먼저** — 이들 접미사를 지명 사전에 추가하고
호모그래프(지구="the earth"·공원=일반명사) 오탐은 기존 stopword 게이팅으로 억제하면,
에어갭·결정적으로 5건 잔여 FN 의 대부분을 커버할 것으로 추정한다(추가 학습·모델 불필요, 최저비용·최고 ROI).
다음으로 **P3(규칙∪NER 유니온)** 은 규칙에 안 걸리는 라틴 혼용(`여의도 IFC`)과 다토큰 행정
경계(§2.2 의 14행 class d)를 백스톱해 span-exact 품질을 확보한다 — P2 만으로는 경계 오차가 남는다.
**P4(조건부 미세조정)** 대상은 규칙·NER 유니온으로도 안 잡히는 진짜 신규 고유지명으로,
현 골드셋 기준 추정 잔여 ≈0 이므로 0.90+ 달성에는 **후순위(대개 불필요)**.
**선결 과제:** 현 골드셋에는 **도로명주소·상세주소 샘플이 0건**이라, 현재 지표는 "지명 인식"만
측정한다. 상위 트랙의 "주소 0.90+" 를 실제 주소 커버리지로 입증하려면 골드셋에 (a)로/길+번지·
(b)동/호·건물명 케이스를 **확장**하는 선행 작업이 P2 와 함께/앞서 필요하다(미확장 시 0.90+ 는
지명 recall 만 증명).
