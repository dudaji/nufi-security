# KR_PERSON 오차 분석 — false-negative 덤프·분류 (인명 정확도 착수 단계)

목적: 공개 README 에 남은 마지막 정확도 한계인 **인명(KR_PERSON)** 개선을 위해, 주소
(KR_LOCATION)에서 통한 플레이북(오차분석 → 규칙/유니온 → 골드셋 → 게이트)의 **첫 단계**
—오차분석—만 수행한다. 프로덕션 모델(onnx-int8)의 인명 재현율 신뢰구간 하한이 목표선을
소폭 하회하는 원인을 골드셋 false-negative 로 귀인하고, 다음 minor(v0.3.0) 인명 캠페인의
우선순위를 데이터로 확정한다. **이 단계는 코드·규칙·모델을 바꾸지 않는 측정·분류 산출물이다.**

- 재현(에어갭·결정적): `python3 scripts/dump_kr_person_fn.py`
- 원자료(JSON): `docs/reports/kr-person-fn-dump.json` (프로덕션 집계·후보 풀·참고 덤프 전량)
- 프로덕션 기준선: `docs/reports/recall-int8.json` (onnx-int8, test split)
- 골드셋: `samples/gold/{test,dev}.jsonl` (전량 합성, CC0). `KR_PERSON` test 126 / dev 84

## 1. 결론 (요약)

- 프로덕션(onnx-int8) 인명 재현율은 **0.9127** (test 115/126)로 점추정은 목표선 0.90 을
  넘지만, **95% 신뢰구간 하한이 0.8504** 라 "통계적으로 확신을 갖고 0.90 이상"이라고는
  아직 말할 수 없다. 이 하한 부족분이 README 가 인명을 마지막 한계로 남긴 이유다.
- 놓친 11건(test) 중 **약 9건이 사전 미수록 인명**, 약 2건이 수록 인명이다(§2 분해).
  즉 오차는 압도적으로 **미수록 인명(어휘밖 성·이름)** 에 집중된다 — 주소가 "어휘밖
  고유지명"에 집중됐던 것과 같은 구조다.
- 미수록 인명 후보 풀(test+dev 118건)을 이름 형태로 나누면 **희성/미수록 단성 78건,
  복성(2음절 성) 40건**. v0.3.0 은 이 두 형태를 겨냥한 **성씨·이름 사전 확장 + 규칙∪NER
  유니온**이 최저비용·최고 ROI 로 추정된다(주소 P2→P3 와 동형).

## 2. 프로덕션 재현율 분해 (권위 수치 — 커밋 baseline)

출처 `docs/reports/recall-int8.json` (onnx-int8, test split, n=126):

| 구분 | 분모 | 적중 | FN | recall | 95% CI |
|---|---:|---:|---:|---|---|
| KR_PERSON 전체 | 126 | 115 | **11** | 0.9127 | 0.8504 – 0.9506 |
| ├ 미수록 인명 | 72 | 63 | **≈9** | 0.875 | 0.7792 – 0.9328 |
| └ 수록 인명(추정) | 54 | 52 | **≈2** | 0.963 | — |

- 미수록/수록 분해는 baseline 의 `person_unlisted_recall=0.875`·`person_unlisted_n=72` 에서
  정수 반올림한 **추정**이다(모델 재실행 시 확정). 방향은 명확하다 — **FN 의 ~82%가
  미수록 인명**.
- 전체 CI 하한(0.8504)을 끌어내리는 실질 동인은 미수록 슬라이스다(그 CI 하한 0.779).
  수록 인명은 recall 0.963 으로 이미 목표선을 넉넉히 넘는다.
- 게이트 관점: 현 수용 기준(`person_recall≥0.85` **및** `person_recall_ci_low≥0.85`)은
  0.8504 로 **간신히 통과**한다. README 가 인용하는 "0.90+ 를 신뢰구간으로 입증"까지는
  미수록 슬라이스의 표본·recall 을 함께 올려야 한다(v0.3.0 범위).

## 3. 프로덕션 FN 후보 풀 — 미수록 인명 전량 분류 (test+dev 118건)

프로덕션 FN 의 행 단위 확정은 모델 재실행이 필요하다(§5). 모델 미가용 환경에서는 FN 이
집중되는 모집단 = **미수록 인명 전량**을 후보 풀로 놓고 형태·도메인·경계를 분류한다.

### 3.1 성씨 형태 (주 신호)

| 형태 클래스 | 건수 | 비중 | 예시 | 왜 어려운가 |
|---|---:|---:|---|---|
| 희성/미수록 단성 | 78 | 66% | 탁·갈·어·견·옹·모·초·봉·범·피·승·복·화·호 | 저빈도 성씨 — 서브워드 NER 의 성씨 사전·통계 신호 약함 |
| 복성(2음절 성) | 40 | 34% | 선우·남궁·황보·제갈·독고·사공·서문·동방 | 성 경계가 2음절 — 모델이 성+이름 분절을 오인하기 쉬움 |

- split 별: test 미수록 72(복성 21·희성 51) / dev 미수록 46(복성 19·희성 27).
- **흔한 단성(김·이·박·최·정 등)은 후보 풀에 없다** — 수록 인명이며 recall 0.963(§2).

### 3.2 도메인 (컨텍스트 분포 — 편중 없음)

미수록 인명은 7개 업무 도메인에 고르게 퍼져 있다(특정 문맥 편중 아님):
부장 21 · 수신자 20 · 예금주 18 · 가입 신청자 16 · 환자 12 · 고객 11 · 선생님/담당자 각 10.
→ 오차 원인은 **문맥이 아니라 이름 형태**임을 시사한다.

### 3.3 조사 경계

이름 직후 조사·경칭이 공백 없이 부착된 케이스 49 / 미부착(공백) 69. 부착·미부착이 모두
상당수라, 조사 경계는 이차적 요인이다(주 신호는 §3.1 성씨 형태).

## 4. 참고: gazetteer 백엔드 FN 덤프 (구조 참고 — 프로덕션 프록시 아님)

에어갭·결정적 재현을 위해 gazetteer 백엔드로도 전량 채점했다. **단, gazetteer 는 사전
매칭 위주라 인명 recall 이 낮다**(test 0.389 / dev 0.345) — 프로덕션(onnx-int8, 학습형 NER,
recall 0.9127) FN 목록의 **프록시가 아니다**. 이 덤프는 이름 형태 분포의 구조 참고치로만
사용한다.

- gazetteer FN(test+dev 132건): 희성 75 · 복성 37 · 흔한 단성 20. 미수록 108 / 수록 24.
- **주의(주소 대비 차이):** 주소에서는 gazetteer FN 이 프로덕션 FN 의 상위집합(동일 클래스
  집중)이라 프록시로 쓸 수 있었다. 인명은 다르다 — gazetteer 는 미수록 인명을 거의 못 잡고,
  나아가 수록 인명 24건도 놓치는데 이는 전부 특정 두 문장 템플릿("수신자 X 앞으로",
  "담당자 X에게")에서 발생하는 **gazetteer 고유의 템플릿 아티팩트**이며 프로덕션 모델에는
  없다. 따라서 gazetteer FN 은 프로덕션 FN 카운트로 **인용하지 말 것**. 권위 수치는 §2.

## 5. 후속 v0.3.0 캠페인 권고 (근거)

프로덕션 인명 FN 은 **미수록 인명(≈82%)** 에 집중되고, 그 형태는 **희성/미수록 단성(66%)
+ 복성(34%)** 이다. 주소에서 검증된 플레이북을 그대로 적용한다:

1. **성씨·이름 사전 확장(규칙, P2 대응)** — 표준 복성 전량(선우·남궁·황보·제갈·독고·
   사공·서문·동방 등)과 저빈도 단성을 성씨 사전에 추가. 흔한 이름 접미 패턴과 결합해
   에어갭·결정적으로 미수록 인명의 상당 부분을 커버. 오탐 억제는 기존 stopword/문맥
   게이팅 재사용. 최저비용·최고 ROI 로 추정.
2. **규칙∪NER 유니온(P3 대응)** — 규칙이 못 잡는 신규·희귀 이름을 학습형 NER 이 백스톱.
   주소에서 확인된 대로 union recall ≥ max(규칙, 모델).
3. **골드셋 미수록 슬라이스 확장(선행)** — 현 미수록 표본(test 72)의 Wilson CI 하한
   (0.779)을 목표선 위로 좁히려면 표본 확대가 필요. 주소에서 쓴 sort-last
   append(sealed 골드셋 불변) 절차를 그대로 사용.
4. **게이트** — `person_recall_ci_low ≥ 0.90` 을 신규 수용 기준으로 승격(현 0.85 에서 상향).

**프로덕션 FN 행 단위 확정(선행 측정)**: 모델 프로비저닝 환경에서
`python3 scripts/dump_kr_person_fn.py --backend onnx-int8` 을 재실행하면 §2 의 추정 9+2 를
행 단위로 확정할 수 있다. 현재 에어갭 환경에는 onnx-int8 이 미프로비저닝이라 §2 는 커밋
baseline 인용으로 대체한다(정직 보고).
