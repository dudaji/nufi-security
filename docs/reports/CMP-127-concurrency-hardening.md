## 0. 한 줄 판정 (binary)

> **수용기준 "부하 하 512자 p95 ≤150ms 를 C≥4 에서 성립"은 단일 프로세스 in-process
> 튜닝(intra-op 캡 + bounded 워커풀)만으로는 달성 불가** — 측정으로 확정. 사유는
> **CPython GIL** 에 의한 공유 추론 파이프라인 직렬화(아래 §3 정량 근거). 대안은
> **프로세스/인스턴스 단위 수평확장**(§4 용량식). 즉 수용기준의 두 번째 분기
> **"미달 시 정량 사유+대안"** 으로 종결한다.

부수 성과(하드닝): onnx-int8 로더가 **양자화 파일을 못 찾아 FP32 모델을 즉석
재export** 하던(=측정 대상이 INT8 이 아니게 되던) **정합성 버그를 수정**했다(§2).

## 1. 측정 환경 (정직 고지)

- **CPU 4코어** WSL2 컨테이너. CMP-123 §3 의 수치(무부하 INT8 52ms, 198ms@C4)는
  **12코어** 환경 — 코어수에 따라 과구독 임계 C 가 이동하므로 절대치는 직접 비교 불가.
- 측정 중 **동시 하트비트 프로세스로 박스가 경합**(베이스라인 C=1 조차 p95>150ms 로
  부풀려짐 ← 단일추론 mean 105ms vs CMP-123 52ms). 따라서 **절대 ms 가 아니라
  (a) 구조적 GIL 근거 (b) 구성 간 상대 비교** 에 결론을 앵커링한다. 운영 K/W 확정과
  수평확장 검증은 **전용(비경합) 박스 재측정**이 필요(§5 후속).

---

## 2. 선결 버그 수정 — INT8 로더 정합성 (`egress_audit/detectors/ner.py`)

CMP-123 이후 optimum 4.5x 환경에서 `_OnnxInt8Backend` 가 두 가지로 오작동했다:

1. **양자화 파일 미지정** → optimum 이 표준 `model.onnx` 를 못 찾고 베이스 모델을
   **즉석 재export(FP32)** → "INT8 측정"이 실제로는 재export 모델 측정. 게다가 library
   자동추론 실패로 **로드 자체가 간헐 실패**(flaky).
2. **디렉터리 해석 버그** — `M5_ONNX_DIR=~/.cache/m5_onnx_int8`(부모명이 `…int8`)일 때
   `endswith("int8")` 오판으로 `int8/` 하위로 못 내려가 `config.json` 부재 에러.

**수정:** `*.onnx` 자동 탐지(`model_quantized.onnx` 우선) + `AutoConfig` 명시 +
`config.json` 유무 기반 견고한 경로 해석. 이로써 **실제 INT8 양자화 모델을 결정적으로
로드**. (이 수정 전의 `CMP-127-before.json` 수치는 신뢰 불가로 폐기.)

## 3. 동시성 스윕 — intra-op 캡 + bounded 워커풀 (in-process)

`NUFI_NER_INTRA_OP_THREADS=K`(세션당 intra-op 캡) + `NUFI_NER_INFER_WORKERS=W`(동시
추론 제한)로 구성. baseline = K=4(코어수)·무제한. 각 구성 동일 입력·동시성으로 측정
(`docs/reports/CMP-127-load-{baseline,k1w4,k2w2,k2w4}.json`).

### p95(ms) — 512자, onnx-int8, 4코어 (✅=≤150ms)

| 구성 | C=1 | C=2 | C=4 | C=8 | C=16 | μ(req/s) | mean(ms) |
|---|---|---|---|---|---|---|---|
| baseline (K=4, 무제한) | 207❌ | 172❌ | 260❌ | 417❌ | 1116❌ | 9.5 | 105 |
| K=1, W=4 | 85✅ | 96✅ | 434❌ | 1485❌ | 3390❌ | 19.0 | 53 |
| K=2, W=2 | 115✅ | 75✅ | 906❌ | 2752❌ | 3847❌ | 8.0 | 125 |
| K=2, W=4 | 38✅ | 85✅ | 275❌ | 599❌ | 1599❌ | 9.1 | 110 |

### 판독

- **캡은 저동시성(C≤2)을 개선**: K=1/W=4·K=2/W=4 가 C=1~2 에서 baseline 대비 p95 를
  크게 낮춤(스레드 과구독·캐시 스래싱 완화). μ 도 K=1/W=4 가 19 req/s 로 최고.
- **그러나 어떤 (K,W) 도 C≥4 에서 ≤150ms 미달.** 결정적 근거: **K=1/W=4** 는
  코어=4, 세션당 1스레드, 동시추론 4로 **이론상 과구독이 0**(총 4스레드=4코어)인데도
  **C=4 p95=434ms** — 만약 4 워커가 진짜 병렬이면 C=4 지연은 단건(~53ms) 근처여야 한다.
  실측이 5~8배 → **워커들이 병렬로 돌지 못함**.
- **원인 = GIL.** 본 풀은 단일 공유 HF `pipeline` 을 세마포어로 감싼 구조다. onnxruntime
  `Run`(C++)은 GIL 을 풀지만, 파이프라인의 **토크나이즈 디스패치 + `aggregation_strategy`
  후처리**(파이썬)가 GIL 을 잡아 **공유 파이프라인 호출이 직렬화**된다. 추론이 빠른 INT8
  일수록 파이썬 글루 비중이 커져 직렬화가 지배적. → **in-process 스레드 풀은 파이썬 핫패스를
  코어 전용화할 수 없다.**

## 4. 대안 — 수평확장 용량식 (구조적 해법)

GIL 회피의 정석은 **스레드가 아니라 프로세스**다. 인스턴스(=프로세스)당 안전
동작점을 정의하고 **N 인스턴스로 수평확장**한다(각 인스턴스 독립 세션/메모리 → GIL 무관).

### 4.1 인스턴스당 안전 동작점
- **C_safe** = p95 ≤150ms 를 만족하는 최대 동시성(로드밸런서 max-inflight 또는
  동시성 리미터로 강제). 측정으로 확정 — CMP-123 계열에서 C_safe ≈ 2.
- **λ_safe** = 큐 안정(λ≤μ)으로 흡수하는 최대 도착률 ≈ **μ**(인스턴스 서비스율,
  `backpressure.service_rate_mu_req_s`). 안전계수 ρ(목표 이용률, 예 0.7): **λ_safe = ρ·μ**.

### 4.2 용량식
```
필요 복제수  N = ⌈ λ_peak / (ρ · μ_instance) ⌉
세이프가드   각 인스턴스 인플라이트 동시성 ≤ C_safe
```
- **μ_instance** 는 전용(비경합) 박스 실측이 필수 — 본 경합 환경의 9~19 req/s 는 하한,
  CMP-123(12코어) 클린 측정은 μ≈22.4(INT8)/13.5(FP32).
- 예) λ_peak=100 req/s, μ=22.4(INT8), ρ=0.7 → N=⌈100/15.7⌉=**7 인스턴스**.
- 예) FP32(정확도 게이트 우선) μ=13.5, 동일 조건 → N=⌈100/9.45⌉=**11 인스턴스**.

### 4.3 배선 (기존 자산 재사용)
- 인스턴스는 무상태(가드/모델만) → 수평 복제 단순. **CMP-122 배포 패키징** 재사용:
  Compose `--scale`/Helm `replicas` 로 N 조정.
- 인스턴스 내부 bounded 풀(K/W)은 코어 과구독을 막아 **μ 를 안정화**(용량식 입력의
  예측가능성 보장) — 즉 in-process 풀의 가치는 "C≥4 p95 게이트 충족"이 아니라
  "인스턴스당 μ 안정화 + C≤2 개선"으로 재정의된다.
- **비동기 감사봇 큐**(FileQueue→AuditBot)는 동기 p95 경로와 분리 — **동적 배치(요청
  합성)**로 인스턴스당 처리량을 키워 λ_safe 를 올릴 수 있다(배치는 지연-처리량
  트레이드오프라 동기 ≤150ms 경로엔 부적합, 비동기 드레인에 적합 → §5 후속).

**메모리 주의(실측):** 본 4코어/저RAM 박스에서 4 프로세스 동시 기동은 OOM-kill — 즉
co-location 밀도도 RAM 제약. 수평확장은 **인스턴스 자원 격리(컨테이너/노드)** 전제.
