# NuFi 보안·증빙 평면 차터 — 한국 특화 PII/DLP 중심

> 상태: **차터 초안 (2026-06-29)** — 통합 NPU AI 플랫폼 내에서 NuFi 가 소유하는 **보안·증빙 평면**의 범위·통합 계약·가드레일을 확정한다. 우선순위·시퀀싱은 [로드맵](../ROADMAP.md)의 채택된 2축(①규제 준수 증빙 게이트웨이 + ②한국어 PII 이그레스 방화벽)을 전제로 한다.
>
> 본 차터는 **차터/스펙 단계**의 산출물이다. 구현 단계는 파트너 서빙 플랫폼의 연동 좌표(인라인 스캐너 훅 인터페이스·트레이스 스키마) 확정 이후 진행한다(§6 블로커).

---

## 1. 포지셔닝 — 단일 플랫폼, 두 평면

통합 제품은 사내 NPU(FuriosaAI RNGD) 위에서 동작하는 **단일 AI 플랫폼**이며, 책임은 두 평면으로 나뉜다.

- **서빙·앱 평면 (파트너 소유):** 모델 서빙, 콘솔/플레이그라운드, 게이트웨이 라우팅·예산·rate limit·metering, 운영 옵저버빌리티, 품질 평가.
- **보안·증빙 평면 (NuFi 소유):** 한국 특화 PII/DLP, 외부 클라우드 이그레스 정책·차단, 변조탐지 해시체인 감사원장, 한국 규제 컴플라이언스 매핑.

> **NuFi = 통합 플랫폼의 「규제 준수·데이터 유출 방지 평면」.** 구매 트리거는 "데이터 유출 걱정"이 아니라 **"혁신금융 지정 부가조건(규제) 준수 증빙"** 이다. 한국어 PII 처리·온프렘·에어갭 — NuFi 고유 자산이 그대로 이 규제 요건과 맞물린다.

두 평면은 같은 사용자 흐름을 공유하되 신뢰 경계 방향이 다르다. 서빙 평면은 사내로 향하는(inward) 모델 제공을, 보안·증빙 평면은 외부 클라우드 경계로 나가는(outward) 데이터 흐름의 통제와 무결성 증빙을 책임진다.

## 2. 범위 — NuFi 소유 (4 기둥)

### 2.1 한국 PII/DLP 엔진 (중심 차별화)

한국 고유 식별자와 개인신용정보를 탐지·차단·가명화한다.

- **대상 식별자:** 주민등록번호, 계좌번호, 사업자등록번호, 여권번호, 운전면허번호, 그리고 개인신용정보.
- **동작:** 입력·출력 양방향 인라인 스캔 → 차단 또는 가명화. 외부 전송 직전 마지막 게이트.
- **차별화 지표(측정 가능):** 식별자 클래스별 탐지 정확도의 신뢰구간 하한(Wilson CI 하한)을 관리 목표로 둔다. 범용 OSS 탐지기로 대체·희석하지 않는다(§4 가드레일).

### 2.2 외부 클라우드 이그레스 정책·차단

외부 LLM(Claude·OpenAI 등)으로 데이터가 전송될 때 한국 PII 유출을 차단한다.

- 채널·목적지·테넌트별 정책 프로파일(다중 프로파일 + 바인딩, 무중단 롤백).
- 정책 위반 시 차단 + 결정 로그 기록. 국외 이전 차단을 직접 충족 통제로 매핑한다.

### 2.3 변조탐지 해시체인 감사원장

규제 증빙용 무결성 게이트.

- 모든 결정·차단·가명화 이벤트를 해시 체인으로 봉인. 변조 시 무결성 게이트가 비-0 종료코드로 실패.
- 운영 트레이스(파트너 옵저버빌리티)와 **공존**한다: 트레이스는 운영 가시성, 감사원장은 규제 증빙 — 입력원은 공유하되 책임은 분리.

### 2.4 한국 규제 컴플라이언스 매핑

금융 AI 가이드라인·망분리 점검항목을 NuFi 통제 커버리지로 자동 산출한다.

- 점검항목 → NuFi 통제 매핑표 + 충족 상태(충족/부분/범위 밖) + 증빙 출처.
- 직접 충족 항목은 기존 리포트 증빙(무결성·차단/가명화 건수·결정 로그)에서 상태를 자동 판정. 커버리지는 정보성이며, 무결성 게이트의 종료코드(0/1)는 유지한다.

## 3. 통합 계약 (파트너 서빙 플랫폼과)

NuFi 평면은 파트너 서빙 평면에 다음 인터페이스로 결합한다. 각 항목의 좌표(스키마·훅 시그니처)는 구현 단계 선행조건이다(§6).

| NuFi 제공 | 결합 지점 (파트너) | 형태 |
|---|---|---|
| 한국 PII/DLP 엔진 | 인라인 가드 스캐너 슬롯 | 슬롯 대체/보강 (인라인 스캐너) |
| 운영 트레이스 → 증빙원장 입력 | 트레이스 파이프라인 | 트레이스 → 감사원장 입력원 |
| 이그레스 정책·차단 | 게이트웨이 요청 훅 | 게이트웨이 훅/플러그인 |
| 컴플라이언스 리포트 | 운영 콘솔 페이지 | 콘솔 내장 페이지 |

원칙: NuFi 는 **자체 IP(한국어 PII 엔진·해시체인 원장·점검항목 매핑)** 를 인터페이스 뒤에 두고, 파트너의 OSS 조립(콘솔·게이트웨이·트레이싱)과 슬롯 단위로 결합한다.

## 4. 가드레일

- **IP 희석 금지:** 한국 PII 엔진의 차별화는 측정 가능한 정확도(CI 하한)와 무결성으로 유지한다. 범용 OSS 탐지기로 대체·약화하지 않는다.
- **에어갭/폐쇄망 단독 납품 유지:** NuFi 코어는 외부 의존 0, 독립 배포 가능한 임베디드 의존성을 유지한다. 통합 제품의 일부로 동작하더라도 코어는 폐쇄망에 단독 납품 가능해야 한다.
- **종료코드 안정성:** 무결성 게이트는 0(정상)/1(변조). 컴플라이언스 커버리지는 정보성이며 신규 비-0 을 도입하지 않는다.

## 5. 범위 밖 (파트너 소유 — NuFi 미관여)

서빙, 콘솔·플레이그라운드, 게이트웨이 라우팅, 예산·rate limit·metering, 운영 옵저버빌리티, 품질 평가(judge·평가 하니스). 추가로 로드맵 §4 의 비초점 통제(판단형 robustness·오염 공격·공급망 SBOM·교육·제3자 독립 검증)는 파트너/이연으로 유지한다.

## 6. 블로커 및 선행조건

- **구현 단계 선행조건:** 파트너 NPUOps 서빙 플랫폼 미구축 → 연동 좌표(인라인 스캐너 훅 인터페이스·트레이스 스키마) 확정 후 진행. **차터/스펙 단계는 선행 가능**(본 문서).
- **확정 대기 (구현 비차단):**
  1. 코드베이스 구조: 모노레포 vs 임베디드 의존성. 권고 = **임베디드 의존성**(에어갭 단독 납품 유지, §4 가드레일 직결).
  2. 통합 제품 오너·로드맵 권한.
  3. 목표(OKR) 연결: 현재 미연결.

## 7. 시퀀싱 (로드맵 연계)

본 차터의 범위는 [로드맵](../ROADMAP.md) §3 우선순위에 그대로 매핑된다.

- **P0 컴플라이언스 매핑 리포트 (S):** §2.4 — 점검표 자동 충족 증빙. 기존 기능 위 증빙 산출이 ROI 최상.
- **P1 요청 횟수 제한 (M):** 서빙 평면(파트너) 게이트웨이와 공동 — NuFi 는 정책·증빙 측면 결합.
- **P2 한국어 회피 공격 인라인 탐지 (L):** §2.1 엔진 확장, 한국어 가드모델 차별화.
- **P3 킬스위치·자산 무결성 (S~M):** §2.3 원장 위 운영 안전장치 격상.

구현 트랙 분해는 §6 의 통합 좌표 확정 이후, 구현 오너(개발자)에게 인계한다. 차터/스펙 단계 산출물(본 문서·인터페이스 명세)은 그 선행 입력이다.
