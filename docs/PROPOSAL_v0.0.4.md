## 한눈에 보기 (요약)

- **v0.0.4의 한 문장 정의:** 운영(Operate) 단계에서 아직 비어 있는 **정책 운영 자동화** 기능을 완성하고, 동시에 그동안 미뤄둔 **정확도 측정 숙제**와 **이미 만들어졌지만 아직 정식 출시되지 않은 패키징 기능**을 함께 마무리하는 버전.
- **왜 지금인가(핵심):** ① 이미 코드로 완성된 설치형 패키징(CMP-141)이 **어느 출시 버전에도 들어가 있지 않아** 사용자가 쓸 수 없는 상태이고, ② 운영 단계의 마지막 기능(정책 운영 자동화)이 빠져 있어 **고객이 여러 워크로드를 동시에 운영할 수 없으며**, ③ 정확도 측정 숙제가 **두 버전 연속(v0.0.1 → v0.0.3) 미뤄져** 유일하게 남은 품질 미달 항목으로 계속 따라다니고 있기 때문입니다.
- **추천 구성:** A0(이미 완성된 패키징 출시) + A1(정책 운영 자동화) = 반드시(Must), A2(정확도 숙제 정리) = 권장(Should), A3(문서 지문 DLP) = 수요 있을 때만(조건부).

## 0. 출발점 — v0.0.3이 무엇을 끝냈고, 무엇을 남겼으며, 지금 무엇이 쌓여 있는가

이 제안을 이해하려면 먼저 직전 버전이 어디까지 왔는지를 정확히 봐야 합니다. 세 덩어리로 나눠 설명합니다.

### (1) 이미 끝낸 것 — v0.0.3 출시 완료 (2026-06-28, 태그 `v0.0.3`)

운영(Operate) 단계의 세 축 중 **두 축**을 출시했습니다.

- **관측(Observe) — 감사 가시성 대시보드(read-only)** (CMP-134): 보안팀이 온프렘(on-prem)에서 *읽기 전용*으로 감사 기록을 보는 화면. 4개 패널(결정 조회 / 해시체인 무결성 / 우회 타임라인 / 카테고리별 추이).
- **보증(Assure) — 상시 커버리지 측정** (CMP-133): "지금 내 트래픽 중 몇 %가 실제로 게이트웨이를 통과하고 있는가"를 항상 집계하는 `nufi-egress coverage`, 그리고 게이트웨이를 우회한 트래픽을 실시간으로 잡아내는 `nufi-egress monitor`.
- 위 두 기능의 1줄 실행 데모와 재현 매뉴얼 (CMP-138).

### (2) 이미 만들어졌지만, 아직 어떤 버전에도 안 들어간 것 ⚠️ (가장 시급)

`v0.0.3` 태그를 찍은 **이후에** 작업이 4개 커밋(commit) 더 쌓였습니다. 현재 로컬 main 브랜치가 원격(origin)보다 4커밋 앞서 있고, 이 작업들은 CHANGELOG의 `[Unreleased]`(미출시) 칸에 머물러 있습니다.

- **CMP-141 — 설치형 패키징(installable packaging)**: `pip install` 후 터미널에서 바로 쓰는 실행 명령(`nufi-egress`, `nufi`)을 정식 진입점(console_scripts)으로 만들었고, 흩어져 있던 감사 조회 기능을 `audit` 하위 명령으로 통합했습니다.
- **CMP-140 — 멀티 프로바이더(multi-provider) 문서화**: 통합 가이드에 Claude/Anthropic·Google·Azure 등 여러 LLM 공급자 지원을 명시하고, 운영 명령 설명을 설치형 `nufi-egress` 기준으로 다시 썼습니다(문서 3커밋).

> **이게 왜 중요한가:** CMP-141은 "설치하면 바로 쓰는 명령어"라는 **사용자가 직접 체감하는 기능**입니다. 그런데 이 기능이 **어떤 출시 태그에도 들어가 있지 않습니다.** 즉 고객 입장에서는 *존재하지 않는 기능*입니다. 이대로 두면 잘 만든 패키징 기능이 무기한 출시되지 않은 채로 방치됩니다.

### (3) 끝내지 못하고 남긴 것 — 이 제안이 겨냥하는 빈자리

- **운영 자동화(Policy Ops)가 비어 있음:** 운영(Operate) 단계의 세 번째 축입니다. v0.0.3에서 일정상 다음으로 미뤘습니다. 현재 정책 변경은 **한 번에 하나씩, 단일 프로파일(profile)만** 무중단 반영됩니다(CMP-124). 여러 프로파일을 동시에 돌리거나, 경로/테넌트별로 다른 정책을 묶거나, 정책 버전을 되돌리는(rollback) 기능이 없습니다. → **고객이 여러 워크로드를 한 게이트웨이로 운영하지 못합니다.**
- **정확도 측정 숙제(이하 'M6 숙제')가 두 버전째 미뤄짐:** INT8 양자화 모델에서 한국어 인명(KR_PERSON) 인식의 신뢰구간(Wilson 신뢰구간) 하한값이 **0.832**로, 목표치 0.85에 약 1.8%포인트 못 미칩니다(표본이 작아 생기는 양자화 노이즈). v0.0.1의 'Known limitations(알려진 한계)'에 적힌 뒤 v0.0.3까지 손대지 못했습니다. **현재 우리 제품에 유일하게 남은 품질 미달 항목**입니다.
- **운영 환경(온프렘) 지연 시간 미측정:** 현재 지연 38ms는 *개발 환경* INT8 측정치입니다. 실제 고객 온프렘 환경의 p95 지연은 아직 재측정하지 못했습니다.
- **문서 지문 DLP(IDM)는 수요 확인 대기:** 디자인 파트너(design partner)의 실제 수요 신호가 있을 때만 착수하기로 v0.0.2·v0.0.3 모두 보류했습니다.

### 정리

v0.0.3이 *"보안팀이 온프렘에서 관측·보증한다"*를 풀었다면, v0.0.4가 풀어야 할 문제는 세 가지입니다 — **운영자가 여러 정책을 안전하게 동시 운영하게 만들고**(운영 자동화 완성), **두 버전 미룬 정확도 숙제를 끝내고**, **이미 완성됐지만 출시되지 않은 패키징을 정식으로 내보내는 것.** 한마디로 v0.0.4는 **운영(Operate) 단계를 끝까지 완성하고, 그동안 쌓인 정리 작업을 마무리하는 버전**입니다. 그다음이 v0.1.0의 확장·차별화(Scale & Differentiate) 단계입니다.

## 1. v0.0.4 후보 트랙(track) 5종 — 이사회가 조합을 고를 수 있게 제시

각 트랙마다 **무엇을 / 왜 지금 / 대표 산출물 / 작업량(Effort) / 담당**을 적습니다. 구현 담당은 엔지니어(Engineer, CMP-96), 단 **출시 권한이 필요한 행위**(태그·버전 확정)는 사람이 맡습니다(CMP-60/137).

작업량 표기: **S**=작음, **M**=보통, **L**=큼, **XL**=매우 큼.

### A0 — 이미 완성된 패키징을 정식 출시 (Release Capture) ★먼저 해야 함
- **무엇을:** 현재 미출시 상태인 CMP-141(설치형 패키징·`audit` 명령 통합)과 CMP-140(멀티 프로바이더 문서)을 v0.0.4의 CHANGELOG 항목으로 정리해 담고, `VERSION`을 0.0.4로 올리고, 출시 태그(`git tag v0.0.4`)를 찍어 원격에 올립니다. (이 트랙만으로도 패치(patch) 버전 하나가 성립합니다.)
- **왜 지금:** 설치형 명령은 사용자가 직접 쓰는 기능인데 **지금 어떤 태그에도 없어 사실상 고객이 못 씁니다.** 새 기능을 그 위에 쌓기 전에, 이미 완성된 것을 먼저 출시해 기준선(baseline)을 고정해야 합니다. 미루면 미룰수록 미출시 변경이 더 쌓여 나중에 한 번에 정리하기 어려워집니다.
- **대표 산출물:** CHANGELOG의 [0.0.4] 항목 초안(이건 CPO가 설계로 제공 가능) + 출시 체크리스트(RELEASE_CHECKLIST) 통과 + 태그 생성(사람이 수행).
- **작업량:** S · **담당:** **사람(릴리스 오너)** — 출시 권한 행위라 AI 미배정(CMP-60/137).

### A1 — 정책 운영 자동화 (Policy Ops at Scale) ★개발의 중심
- **무엇을:** 현재 "한 번에 하나씩" 수준인 정책 반영(CMP-124)을 운영 규모로 끌어올립니다. ① **여러 프로파일 동시 운영** + **경로/테넌트별로 다른 프로파일 묶기(binding)** ② 정책의 **버전 관리·변경 추적·되돌리기(rollback)** 를 무중단으로 ③ **변경 감사 기록**(누가·언제·무엇을 바꿨는지).
- **왜 지금:** 이게 운영(Operate) 단계의 **마지막 남은 기능**입니다. 단일 프로파일·단건 반영만으로는 고객이 부서·서비스·테넌트별로 다른 정책을 한 게이트웨이에서 운영할 수 없습니다. 실제 도입 고객이 가장 먼저 부딪히는 운영 벽이며, v0.0.3에서 의도적으로 다음으로 미뤄둔 바로 그 작업입니다. 지금 채워야 운영(Operate) 단계가 완성됩니다.
- **대표 산출물:** 프로파일 레지스트리(registry) + 묶기 규칙(`routing.yaml` 확장), 버전·되돌리기 명령(`nufi-egress policy …`), 변경 감사 로그.
- **작업량:** M · **담당:** 엔지니어(Engineer, CMP-96).

### A2 — 정확도 측정 숙제 끝내기 (Accuracy Debt, 일명 'M6')
- **무엇을:** ① 기반 인식 모델(NER base model)을 격상해 **INT8 모델의 한국어 인명 신뢰구간 하한을 0.85 이상**으로 올립니다(현재 0.832). ② **실제 온프렘 환경의 p95 지연을 재측정**합니다(현재 38ms는 개발 환경 수치). 모델 격상 시 INT8 재양자화(requantization) 정합성도 함께 검증합니다.
- **왜 지금:** 이 항목은 **두 버전 연속(v0.0.1 → v0.0.3) 미뤄진 유일한 품질 미달 지표**입니다. 끝내면 v0.0.1의 'Known limitations(알려진 한계)'에서 가장 핵심 항목이 사라져, "한국어 PII 재현율 ≥ 0.9 / 지연 p95 ≤ 150ms"라는 우리 핵심 품질 약속이 신뢰구간 하한까지 깔끔하게 충족됩니다. 운영 자동화(A1)를 고객에게 내보낼 때 "정확도도 마무리됐다"고 함께 말할 수 있어야 설득력이 큽니다. 단, 모델 격상은 평가·재양자화 비용이 있어 작업량 변동이 있습니다.
- **대표 산출물:** 격상 모델 벤치마크 리포트(기존 M5 측정 하니스 재사용) + INT8 정합성 테스트 + 온프렘 p95 측정표.
- **작업량:** M–L · **담당:** 엔지니어(Engineer, CMP-96).

### A3 — 탐지 깊이 확장 / 문서 지문 DLP (IDM) — 수요 있을 때만
- **무엇을:** 기밀 탐지의 3번째 계층인 **문서 지문(IDM, Indexed Document Matching)** — 미리 색인해 둔 사내 문서와 유출 시도 내용을 대조해 잡아내는 방식(기존 EDM의 비구조화 확장). 임베딩(embedding) 기반 분류기는 그다음 단계.
- **왜 (지금은 아닐 수 있음):** 차별화된 DLP(Data Loss Prevention, 정보 유출 방지) 기능이지만, **디자인 파트너의 실제 수요 신호에 의존**합니다. 수요가 확인되지 않은 상태에서 먼저 투자하는 것은 권하지 않습니다(v0.0.2·v0.0.3 모두 같은 이유로 보류).
- **대표 산출물:** (수요 확인 시 별도 설계) 문서 색인기(indexer) + 대조기(matcher).
- **작업량:** L–XL · **담당:** 엔지니어(수요 확인 후).

### A4 — 출시 자동화 (Release Automation) — 위생 후속, 이번엔 보류 권장
- **무엇을:** 지금 매번 손으로 하는 VERSION·CHANGELOG·태그·체크리스트 작업을 스크립트로 묶습니다(드라이런(dry-run) + 검증 게이트 포함).
- **왜 (지금은 아님):** v0.0.2~0.0.4 출시가 매번 수작업이라 자동화하면 정리 부담이 줄어듭니다. 다만 출시를 자동으로 찍는 도구는 **출시 권한에 맞닿아 있어** 설계는 CPO가 할 수 있어도 구현·실행은 사람 게이트가 필요합니다. 우선순위가 낮아 v0.1.0으로 미루기를 권합니다.
- **대표 산출물:** `scripts/release.sh`(드라이런) + 체크리스트 자동 검증.
- **작업량:** S–M · **담당:** 사람(출시 권한 인접).

## 2. 로드맵 갱신 (제품 단계 = Horizon)

```
            │ 채택(Adopt)      │ 운영(Operate)            │ 운영 완성(Operate 마무리)        │ 확장·차별화(Scale)
```

## 3. v0.0.4 추천 구성 (CPO 추천 · 우선순위 MoSCoW)

| 우선순위 | 트랙 | 추천 근거 |
|---|---|---|
| **반드시(Must)·먼저** | **A0** 설치형 패키징 정식 출시(CMP-141/140) | 이미 완성된 사용자 대면 기능이 미출시 상태. 기준선부터 고정 |
| **반드시(Must)** | **A1** 정책 운영 자동화(O3 이월) | 운영(Operate)의 마지막 기능. 다중 워크로드 운영의 핵심 |
| **권장(Should)** | **A2** 정확도 측정 숙제 끝내기(M6) | 두 버전 미룬 유일한 품질 미달 지표 정리 |

**출시 순서 — 두 가지 선택지(이사회 결정, §5의 2번):**
- **선택지 ⓐ 묶어서 출시(권장):** 미출시분(A0 내용)을 v0.0.4에 *함께* 담고, A1(+권장 A2) 개발이 끝나면 **한 번에 v0.0.4 태그**를 찍습니다. 출시 1회, CMP-141 패키징이 새 기능과 같이 나갑니다.
- **선택지 ⓑ 먼저 패치, 나중에 기능:** CMP-141/140만 **즉시 v0.0.4 패치로 출시**하고(A0), 정책 운영 자동화(A1)·정확도 숙제(A2)는 **v0.0.5**로 돌립니다. 설치형 패키징을 고객 손에 더 빨리 쥐어줍니다.

각 마일스톤(milestone)의 완료 기준(이분법으로 명확히):
- **M0(A0):** `VERSION`=0.0.4 · CHANGELOG에 [0.0.4] 항목 존재(CMP-140/141 + 채택 트랙) · `git tag v0.0.4` 원격 반영 완료(사람 릴리스 오너).
- **M1(A1):** 프로파일 2개를 동시에 운영 + 경로별 묶기(binding) 1건 적용 · 정책 버전 되돌리기(rollback) 1건을 무중단으로 수행 · 변경 감사 로그 1건 기록.
- **M2(A2, 권장):** 격상 모델로 INT8 한국어 인명 신뢰구간 하한 ≥ 0.85 재현 · 온프렘 p95 측정표 1건 산출.

## 5. 이사회/CEO 정렬이 필요한 결정 사항

1. **v0.0.4 채택 여부 + 트랙 구성** — 추천안(A0 먼저·Must, A1 Must, A2 Should, A3 조건부)을 채택하시겠습니까? (새 버전 라인은 이사회 유보 사항, CMP-137.)
2. **출시 순서** — 선택지 ⓐ 묶어서 1회 출시(권장) vs ⓑ CMP-141 즉시 패치 후 운영 자동화는 v0.0.5?
3. **디자인 파트너 수요 상태(A3 게이트)** — 문서 지문/DLP 수요 신호가 있습니까? 없으면 A3는 이번에 안 함 유지.
4. **A2 정확도 숙제 범위** — 기반 모델 격상에 착수해도 됩니까? 온프렘 p95 측정용 환경을 제공할 수 있습니까?
5. **A0 릴리스 오너 지명** — 누가·언제 v0.0.4 태그를 찍습니까? **구현·권한 행위라 사람 담당이 필수입니다(CMP-60/137).** CPO는 CHANGELOG 초안 설계만 제공합니다.
6. **OKR 연결 🚩** — 이 이슈의 목표(goalId)를 확인해야 합니다. v0.0.4를 연결할 회사 목표/핵심결과(Objective/KR)가 없다면 'NuFi 운영→확장' 목표를 갱신합니다.
