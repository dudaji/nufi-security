## 한눈에 보기 (요약)

- **v0.0.5의 한 문장 정의:** v0.0.4가 "설치하면 하나의 CLI(`nufi-egress`)로 운영한다"는 **도입(adoption) 표면**을 닦았으니, v0.0.5는 그 위에서 **정책을 실제로 규모 있게 운영하는 기능**을 채우고, 동시에 **세 버전째 미뤄둔 정확도 측정 숙제**를 끝냅니다.
- **왜 지금인가(핵심):** ① v0.0.4로 CLI·패키징은 좋아졌지만, 정작 그 CLI로 굴리는 정책 운영은 아직 **단일 프로파일·한 번에 하나씩**이라 **고객이 여러 워크로드를 한 게이트웨이로 운영하지 못합니다.** 도입 다음에 바로 부딪히는 운영 병목입니다. ② 정확도 숙제(INT8 한국어 인명 신뢰구간 하한 0.832 < 0.85)는 이제 **세 버전(v0.0.1 → 0.0.3 → 0.0.4) 연속 미뤄진** 유일한 품질 미달 항목이고, v0.0.4가 "새 기능 없음" 패치였던 만큼 **지금이 품질 숙제를 끝낼 적기**입니다.
- **추천 구성:** B1(정책 운영 자동화) = 반드시(Must)·이번 버전의 중심, B2(정확도 숙제 끝내기) = 권장(Should), B3(문서 지문 DLP) = 수요 있을 때만(조건부).

## 0. 출발점 — v0.0.4가 무엇을 했고, v0.0.5에 무엇이 남았는가

### (1) v0.0.4 = 도입성 패치 (현재 릴리스 오너가 진행 중)

v0.0.4는 **새 기능이 없는** 도입성(adoption) 패치입니다. v0.0.3(관측 O1·보증 O2) 위에 *이미 완성돼 있던* 작업을 정식 출시로 묶은 것입니다. 운영 동작·정책·탐지 코어는 그대로이고, 레거시 진입점도 전부 하위호환으로 유지됩니다.

- **설치형 콘솔 스크립트(console_scripts) (CMP-141):** `pip install -e .` 후 `nufi-egress`(별칭 `nufi`)를 터미널에서 바로 실행.
- **통합 CLI 마감 (CMP-141·143):** 감사 봇·로그 조회를 `nufi-egress audit` 로, 캡처 레이어 운영 명령을 `nufi-egress targets`·`flow-tap` 으로 흡수 — 흩어져 있던 `python3 -m …` 명령을 하나의 CLI로 통합(마지막 통합 잔여 작업 마감).
- **입문 튜토리얼 (CMP-143):** 토이 프로젝트로 SDK 한 줄 전환 + CLI 운영을 끝까지 실습하는 `docs/HANDS_ON.md`.
- **멀티 프로바이더(multi-provider) 문서화 (CMP-140):** Anthropic/Google/Azure 등 여러 LLM 공급자 지원 명시.

> **CPO 리뷰 메모(CMP-143):** 통합 CLI 코드(`enforcement/cli.py`)를 직접 검증했습니다 — 새 `targets`/`flow-tap` 명령의 출력이 레거시 `python3 -m capture.*` 와 **완전히 동일**(parity 확인), 인자 없는 `flow-tap` 은 종료코드 2, `--simulate` 리플레이가 우회 2건을 정상 탐지. 기존 명령 무회귀. **리뷰 결과: 통과(PASS).** 이 작업은 v0.0.4 도입성 패치에 적절히 담겼습니다.

### (2) v0.0.5가 채워야 할 빈자리 — 운영을 '기능'으로 완성

v0.0.4가 도입 표면(설치·CLI·문서)을 닦았다면, 정작 그 CLI로 굴릴 **운영 기능 자체**는 아직 얕습니다. 두 가지가 남았습니다.

- **운영 자동화(Policy Ops)가 비어 있음 — 운영(Operate) 단계의 마지막 축:** 현재 정책 변경은 **한 번에 하나씩, 단일 프로파일(profile)만** 무중단 반영됩니다(CMP-124). 여러 프로파일을 동시에 돌리거나, 경로/테넌트별로 다른 정책을 묶거나(binding), 정책 버전을 되돌리는(rollback) 기능이 없습니다. → **고객이 여러 워크로드를 한 게이트웨이로 운영하지 못합니다.** CLI는 좋아졌는데 그 CLI로 할 운영이 빈약한, 도입 직후의 전형적 병목입니다.
- **정확도 측정 숙제(이하 'M6 숙제')가 세 버전째 미뤄짐:** INT8 양자화 모델에서 한국어 인명(KR_PERSON) 인식의 신뢰구간(Wilson 신뢰구간) 하한값이 **0.832**로, 목표치 0.85에 약 1.8%포인트 못 미칩니다(표본이 작아 생기는 양자화 노이즈). v0.0.1의 'Known limitations(알려진 한계)'에 적힌 뒤 v0.0.3·v0.0.4까지 손대지 못했습니다. **현재 우리 제품에 유일하게 남은 품질 미달 항목**이며, 함께 묶인 **온프렘(on-prem) p95 지연 재측정**(현재 38ms는 개발 환경 수치)도 미완입니다.
- **(보류 유지) 문서 지문 DLP(IDM):** 디자인 파트너(design partner)의 실제 수요 신호가 있을 때만 착수 — v0.0.2~0.0.4 모두 같은 이유로 보류.

### 정리

v0.0.4가 *"설치하면 하나의 CLI로 운영한다"*를 풀었다면, v0.0.5가 풀 문제는 두 가지입니다 — **운영자가 여러 정책을 안전하게 동시 운영하게 만들고**(운영 자동화 완성), **세 버전 미룬 정확도 숙제를 끝내는 것.** 한마디로 v0.0.5는 **운영(Operate) 단계를 기능으로 끝까지 완성하는 버전**입니다. 그다음이 v0.1.0의 확장·차별화(Scale & Differentiate) 단계입니다.

## 1. v0.0.5 후보 트랙(track) — 이사회가 조합을 고를 수 있게 제시

각 트랙마다 **무엇을 / 왜 지금 / 대표 산출물 / 작업량(Effort) / 담당**을 적습니다. 구현 담당은 엔지니어(Engineer, CMP-96), 단 **출시 권한이 필요한 행위**(태그·버전 확정)는 사람이 맡습니다(CMP-60/137). 작업량: **S**=작음, **M**=보통, **L**=큼, **XL**=매우 큼.

### B1 — 정책 운영 자동화 (Policy Ops at Scale) ★이번 버전의 중심
- **무엇을:** 현재 "한 번에 하나씩" 수준인 정책 반영(CMP-124)을 운영 규모로 끌어올립니다. ① **여러 프로파일 동시 운영** + **경로/테넌트별로 다른 프로파일 묶기(binding)** ② 정책의 **버전 관리·변경 추적·되돌리기(rollback)** 를 무중단으로 ③ **변경 감사 기록**(누가·언제·무엇을 바꿨는지).
- **왜 지금:** 운영(Operate) 단계의 **마지막 남은 기능**입니다. v0.0.4로 단일 CLI 운영까지 왔지만, 단일 프로파일·단건 반영만으로는 고객이 부서·서비스·테넌트별로 다른 정책을 한 게이트웨이에서 운영할 수 없습니다. 도입(adoption) 바로 다음에 부딪히는 운영 벽이며, 이걸 채워야 운영 단계가 완성됩니다.
- **대표 산출물:** 프로파일 레지스트리(registry) + 묶기 규칙(`routing.yaml` 확장), 버전·되돌리기 명령(`nufi-egress policy …`), 변경 감사 로그.
- **작업량:** M · **담당:** 엔지니어(Engineer, CMP-96).

### B2 — 정확도 측정 숙제 끝내기 (Accuracy Debt, 일명 'M6')
- **무엇을:** ① 기반 인식 모델(NER base model)을 격상해 **INT8 모델의 한국어 인명 신뢰구간 하한을 0.85 이상**으로 올립니다(현재 0.832). ② **실제 온프렘 환경의 p95 지연을 재측정**합니다(현재 38ms는 개발 환경 수치). 모델 격상 시 INT8 재양자화(requantization) 정합성도 함께 검증합니다.
- **왜 지금:** **세 버전 연속(v0.0.1 → 0.0.3 → 0.0.4) 미뤄진 유일한 품질 미달 지표**입니다. v0.0.4가 "새 기능 없음" 도입성 패치였던 만큼, v0.0.5에서 이 숙제를 끝내면 v0.0.1의 'Known limitations(알려진 한계)'에서 핵심 항목이 사라져 "한국어 PII 재현율 ≥ 0.9 / 지연 p95 ≤ 150ms" 핵심 품질 약속이 신뢰구간 하한까지 깔끔히 충족됩니다. 단, 모델 격상은 평가·재양자화 비용이 있어 작업량 변동이 있습니다.
- **대표 산출물:** 격상 모델 벤치마크 리포트(기존 M5 측정 하니스 재사용) + INT8 정합성 테스트 + 온프렘 p95 측정표.
- **작업량:** M–L · **담당:** 엔지니어(Engineer, CMP-96).

### B3 — 탐지 깊이 확장 / 문서 지문 DLP (IDM) — 수요 있을 때만
- **무엇을:** 기밀 탐지의 3번째 계층인 **문서 지문(IDM, Indexed Document Matching)** — 미리 색인해 둔 사내 문서와 유출 시도 내용을 대조해 잡아내는 방식(기존 EDM의 비구조화 확장). 임베딩(embedding) 기반 분류기는 그다음 단계.
- **왜 (지금은 아닐 수 있음):** 차별화된 DLP(Data Loss Prevention, 정보 유출 방지) 기능이지만, **디자인 파트너의 실제 수요 신호에 의존**합니다. 수요가 확인되지 않은 상태에서 먼저 투자하는 것은 권하지 않습니다(v0.0.2~0.0.4 모두 같은 이유로 보류).
- **대표 산출물:** (수요 확인 시 별도 설계) 문서 색인기(indexer) + 대조기(matcher).
- **작업량:** L–XL · **담당:** 엔지니어(수요 확인 후).

## 2. 로드맵 갱신 (제품 단계 = Horizon)

```
            │ 채택(Adopt)      │ 운영(Operate)            │ 운영 완성              │ 확장·차별화(Scale)
            │ v0.0.2 ✅        │ v0.0.3 ✅ + v0.0.4(도입성)│ v0.0.5 ← 본 제안       │ v0.1.0
────────────┼──────────────────┼──────────────────────────┼────────────────────────┼─────────────────────
 관측        │ 감사 적재만      │ O1 대시보드(읽기전용) ✅ │ —                      │ 멀티테넌트 콘솔
 보증        │ doctor 1회 진단  │ O2 커버리지 측정·알림 ✅ │ —                      │ SLA 리포팅
 운영 자동화 │ 단건 반영        │ —(미완)                  │ B1 정책 운영 자동화    │ 멀티테넌시·권한관리(RBAC)
 정확도      │ —               │ —(M6 미착수)             │ B2 측정 숙제 끝내기    │ —
 패키징·도입 │ nufi-egress CLI  │ v0.0.4 설치형·통합CLI·입문│ —(완료)                │ 출시 자동화
 탐지 깊이   │ —(수요 관망)     │ —(수요 관망)             │ B3 문서 지문(수요 시)  │ 임베딩 분류기(수요 시)
────────────┴──────────────────┴──────────────────────────┴────────────────────────┴─────────────────────
 버전 주제   │ "반나절에 도입"  │ "온프렘 관측·보증·쉬운도입"│ "정책을 규모있게 운영 + 숙제 정리"│ "다고객·차별화 DLP·SLA"
```

제품 단계는 순서일 뿐 마감일이 아닙니다. B3(탐지 깊이)는 디자인 파트너 수요 신호가 뜨면 어느 단계로든 앞당길 수 있습니다.

## 3. v0.0.5 추천 구성 (CPO 추천 · 우선순위 MoSCoW)

| 우선순위 | 트랙 | 추천 근거 |
|---|---|---|
| **반드시(Must)·중심** | **B1** 정책 운영 자동화 | 운영(Operate)의 마지막 기능. 다중 워크로드 운영의 핵심. 도입 직후의 운영 병목 |
| **권장(Should)** | **B2** 정확도 측정 숙제 끝내기(M6) | 세 버전 미룬 유일한 품질 미달 지표 정리. "새 기능 없음" v0.0.4 직후가 적기 |
| **여유 되면(Could)·조건부** | **B3** 문서 지문 DLP | 디자인 파트너 수요 확인 시 앞당김 |
| **이번엔 안 함(Won't)** | 멀티테넌시 · 권한관리(RBAC) · TLS 중간자(MITM) · 임베딩 분류기 · 출시 자동화 | 범위·리스크·권한 — v0.1.0(확장)으로 |

각 마일스톤(milestone)의 완료 기준(이분법으로 명확히):
- **M1(B1):** 프로파일 2개를 동시에 운영 + 경로별 묶기(binding) 1건 적용 · 정책 버전 되돌리기(rollback) 1건을 무중단으로 수행 · 변경 감사 로그 1건 기록.
- **M2(B2, 권장):** 격상 모델로 INT8 한국어 인명 신뢰구간 하한 ≥ 0.85 재현 · 온프렘 p95 측정표 1건 산출.
- **출시(release):** B1(+권장 B2) 완료 후 `VERSION`=0.0.5 · CHANGELOG `[0.0.5]` 항목 · `git tag v0.0.5` — **사람 릴리스 오너**가 수행(CMP-60/137).

## 4. 리스크(risk)와 완화책

## 5. 이사회/CEO 정렬이 필요한 결정 사항

1. **v0.0.5 채택 여부 + 트랙 구성** — 추천안(B1 Must·중심, B2 Should, B3 조건부)을 채택하시겠습니까? (새 버전 라인은 이사회 유보 사항, CMP-137.)
2. **디자인 파트너 수요 상태(B3 게이트)** — 문서 지문/DLP 수요 신호가 있습니까? 없으면 B3는 이번에 안 함 유지.
3. **B2 정확도 숙제 범위** — 기반 모델 격상에 착수해도 됩니까? 온프렘 p95 측정용 환경을 제공할 수 있습니까?
4. **릴리스 오너 + 순서** — v0.0.4 태그를 먼저 닫은 뒤 v0.0.5 개발을 착수하는 순서에 동의하십니까? v0.0.5 태그를 찍을 사람 릴리스 오너는 누구입니까?(CMP-60/137)
5. **OKR 연결 🚩** — 이 이슈의 목표(goalId)가 아직 비어 있습니다(null). v0.0.5를 연결할 회사 목표/핵심결과(Objective/KR)를 확인하거나 'NuFi 운영→확장' 목표를 갱신해야 합니다.
