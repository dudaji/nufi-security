## 한눈에 보기 (요약)

- **v0.0.6의 한 문장 정의:** 운영이 기능으로 완성됐으니(v0.0.5), v0.0.6은 **"우리 게이트웨이가 제 역할을 하고 있음을 증명"** 하고(신뢰·구매 근거), 동시에 **다고객 확장의 기반을 한 칸 깐다.**
- **왜 지금인가(핵심):** ① 제품은 이제 설치·운영이 다 되지만, 보안·규정준수 구매자(보안담당·감사관)가 결정에 쓰는 **"증명 가능한 SLA/리포트"가 없습니다.** 측정 인프라(O1/O2 대시보드·M5 벤치 하니스)는 이미 있어 **저비용으로 차별화**할 수 있는 빈자리입니다. ② v0.0.5 B1이 **경로/테넌트별 정책 묶기**까지 왔으므로, 다음 운영 벽인 **테넌트 격리·접근제어(RBAC)** 의 첫 슬라이스를 지금 깔면 v0.1.0 확장으로 매끄럽게 이어집니다. ③ 유일하게 남은 측정 숙제인 **온프렘 p95(CMP-147)** 는 러너·런북이 이미 준비돼 있어, 하드웨어만 주어지면 **한 명령으로 종결**됩니다.
- **추천 구성:** **C1(SLA·규정준수 리포팅) = 반드시(Must)·이번 버전의 중심**, **C3(온프렘 p95 종결) = 권장(Should, 하드웨어 게이트)**, **C2(멀티테넌시·RBAC 첫 슬라이스) = 여유 되면(Could)·v0.1.0 헤드라인의 선행**, **C4(문서 지문 DLP) = 수요 있을 때만(조건부)**.

## 0. 출발점 — v0.0.5가 끝낸 것과 v0.0.6에 남은 것

### (1) v0.0.5 = 운영(Operate)을 '기능'으로 완성 (출시 완료)

| 축 | v0.0.5에서 끝낸 것 |
|---|---|
| 운영 자동화 | B1(CMP-144) — 멀티프로파일 동시 운영 + 경로/테넌트별 묶기(binding) + 무재기동 되돌리기(rollback) + 변경 감사 해시체인 |
| 정확도 | B2(CMP-145) — per-channel INT8 재양자화로 KR_PERSON Wilson CI 하한 **0.832→0.850 (≥0.85)** 달성 |
| 관측·보증 | (v0.0.3) O1 읽기전용 대시보드 + O2 상시 커버리지/우회 모니터링 |
| 도입·패키징 | (v0.0.4) `nufi-egress` 설치형 CLI + 통합 + Hands-on 입문 |

→ **설치·운영·정책·정확도가 모두 채워졌습니다.** 다음은 "잘 굴러가고 있음을 증명"하고 "여러 고객으로 확장"하는 단계입니다.

### (2) v0.0.6가 채워야 할 빈자리

- **신뢰·구매 근거(SLA 증명)가 없음 — 차별화의 빈자리:** 우리는 이미 PII recall·지연 p95·커버리지·정책 변경을 *측정*하지만, 그것을 **기간별 SLA/규정준수 리포트로 묶어 보여주는 산출물이 없습니다.** 보안·규정준수 시장의 구매·갱신 결정은 "증명 가능한 SLA"에 달려 있습니다. 측정 인프라가 이미 있어 **가장 저비용·고가치**인 빈자리입니다.
- **다고객 확장의 기반(테넌트 격리·RBAC)이 아직 없음 — 확장의 빈자리:** v0.0.5 B1이 경로/테넌트별 *묶기*까지 왔지만, 테넌트별 **감사로그·매핑(Vault)·설정 격리** 와 **누가 어떤 정책을 보고/바꾸나(RBAC)** 는 아직 없습니다(B1에서 명시적으로 v0.1.0으로 미룸). 완전한 멀티테넌시는 XL이라 v0.1.0 헤드라인이지만, **첫 슬라이스(테넌트 범위 읽기 경계 + 읽기전용 역할)** 는 패치 크기로 선행 가능합니다.
- **마지막 측정 숙제(온프렘 p95)가 하드웨어 대기 중:** CMP-147 — 러너(`scripts/measure_onprem_p95.sh`)·런북·dev-env 인터림표까지 완성, **프로덕션 온프렘 하드웨어만 주어지면 1-명령으로 종결**. v0.0.5에서 분리추적으로 남긴 유일한 미완 측정.
- **(보류 유지) 문서 지문 DLP(IDM):** 디자인 파트너 실제 수요 신호가 있을 때만 — v0.0.2~0.0.5 모두 동일 사유로 보류.

## 1. v0.0.6 후보 트랙(track) — 이사회가 조합을 고를 수 있게 제시

각 트랙마다 **무엇을 / 왜 지금 / 대표 산출물 / 작업량 / 담당**. 구현 담당은 엔지니어(Engineer, CMP-96), 출시 권한 행위(태그·버전 확정)는 사람(CMP-60/137). 작업량: **S**=작음, **M**=보통, **L**=큼, **XL**=매우 큼.

### C1 — SLA·규정준수 리포팅 (Compliance & SLA Reporting) ★이번 버전의 중심
- **무엇을:** 이미 측정 중인 지표를 **기간별 리포트 산출물**로 묶습니다. ① **SLA 리포트** — PII recall SLA, 지연 p95 SLA, 커버리지 SLA를 기간(일/주/월)별로 집계·판정(목표 대비 충족/위반). ② **규정준수 요약** — 정책 변경 감사(누가·언제·무엇을 + 해시체인 무결성), 차단/가명화 건수, 우회 탐지 요약. ③ 산출물 형식 — `nufi-egress report` 로 HTML/Markdown(+JSON) 리포트 생성, 감사관·구매자에게 그대로 제출 가능.
- **왜 지금:** 보안·규정준수 제품의 **구매·갱신 결정 = 증명 가능한 SLA.** 측정 인프라(O1/O2 대시보드·M5 벤치 하니스·감사 해시체인)가 **이미 다 있어** 새 측정 없이 묶기만 하면 됩니다 — 가장 높은 가치/노력 비율. 현재 단일테넌트 온프렘 설치 기반 **모든 고객에게 즉시 유효**.
- **대표 산출물:** `nufi-egress report {sla,compliance}` 서브커맨드 + 리포트 템플릿 + 1-명령 데모 + 검증 테스트. 기존 대시보드 어댑터·M5 리포트 JSON 재사용.
- **작업량:** **M** · **담당:** 엔지니어(Engineer, CMP-96).

### C2 — 멀티테넌시·RBAC 첫 슬라이스 (Scale 기반 선행)
- **무엇을:** 완전한 멀티테넌시(v0.1.0)의 **안전한 첫 칸**만 깝니다. ① **테넌트 범위 읽기 경계** — 감사로그·리포트 조회를 테넌트별로 필터·격리(한 테넌트가 다른 테넌트 로그를 못 봄). ② **읽기전용 역할(RBAC)** — viewer/operator 역할 구분으로 정책 변경 권한과 조회 권한 분리. ③ 기존 B1 정책 묶기(binding)의 테넌트 키를 격리 경계로 승격.
- **왜 (선택적으로) 지금:** B1이 테넌트별 묶기까지 와 있어 **다음 자연스러운 칸**입니다. 다만 완전한 테넌트 격리(Vault·런타임 분리)·쓰기 RBAC·권한 위임은 **XL = v0.1.0 헤드라인**이므로, v0.0.6에서는 **읽기 경계 + 읽기전용 역할 MVP**로만 한정해 범위 폭주를 막습니다.
- **대표 산출물:** 테넌트 범위 필터 + viewer/operator 역할 게이트 + 격리 테스트. (전체 멀티테넌시는 v0.1.0 별도 설계.)
- **작업량:** **M–L** · **담당:** 엔지니어(Engineer, CMP-96).

### C3 — 온프렘 p95 측정 숙제 종결 (CMP-147)
- **무엇을:** v0.0.5에서 분리추적으로 남긴 **프로덕션 온프렘 p95 지연 재측정**을 완료해 M6 정확도/지연 숙제를 ②까지 완전 종결. 러너·런북·dev-env 인터림표는 이미 완성(`b7374d1`).
- **왜 지금:** 유일하게 남은 측정 숙제. 작업량은 작지만 **프로덕션 온프렘 하드웨어가 있어야** 실행됩니다(현재 BLOCKED). 하드웨어가 주어지면 `HOST_CLASS=production … scripts/measure_onprem_p95.sh` **한 명령으로 종결**.
- **대표 산출물:** 온프렘 p95 측정표(호스트 프로비넌스 자기라벨) + CMP-147 종결.
- **작업량:** **S** (러너 완성) · **담당:** 엔지니어 · **게이트:** 이사회/CEO가 온프렘 하드웨어 제공(미제공 시 계속 분리추적).

### C4 — 탐지 깊이 확장 / 문서 지문 DLP (IDM) — 수요 있을 때만
- **무엇을:** 기밀 탐지 3번째 계층인 문서 지문(IDM, Indexed Document Matching) — 색인한 사내 문서와 유출 시도를 대조(EDM 비구조화 확장). 임베딩 분류기는 그다음.
- **왜 (지금은 아닐 수 있음):** 차별화된 DLP이지만 **디자인 파트너 실제 수요 신호에 의존**. 수요 미확인 상태 선투자는 비권장(v0.0.2~0.0.5 동일).
- **대표 산출물:** (수요 확인 시 별도 설계) 문서 색인기 + 대조기.
- **작업량:** **L–XL** · **담당:** 엔지니어(수요 확인 후).

## 3. v0.0.6 추천 구성 (CPO 추천 · 우선순위 MoSCoW)

| 우선순위 | 트랙 | 추천 근거 (가치/노력 + 단계 정렬) |
|---|---|---|
| **반드시(Must)·중심** | **C1** SLA·규정준수 리포팅 | 측정 인프라가 이미 있어 **가치/노력 비율 최상**. 보안·규정준수 구매·갱신의 핵심 근거. 현재 설치 기반 전체에 즉시 유효 |
| **권장(Should)·게이트** | **C3** 온프렘 p95 종결 | 유일하게 남은 측정 숙제. 작업량 S(러너 완성). **하드웨어 제공 시** 한 명령 종결, 미제공 시 분리추적 유지 |
| **여유 되면(Could)** | **C2** 멀티테넌시·RBAC 첫 슬라이스 | v0.1.0 헤드라인의 선행 칸. B1 묶기 다음 자연스러운 단계. 단 읽기경계+읽기전용 역할 MVP로만 한정 |
| **수요 시(조건부)** | **C4** 문서 지문 DLP | 디자인 파트너 수요 확인 시 앞당김 |
| **이번엔 안 함(Won't)** | 완전 멀티테넌시 · 쓰기 RBAC·권한 위임 · TLS 중간자(MITM) · 임베딩 분류기 · 실시간 SLA 알림·콘솔 · 출시 자동화 | 범위·리스크·권한 — v0.1.0(확장)으로 |

각 마일스톤 완료 기준(이분법으로 명확히):
- **M1 (C1, Must):** `nufi-egress report sla` 1건 + `report compliance` 1건이 **샘플 감사로그로부터 HTML/MD 리포트를 생성**하고, 각 SLA 항목(recall·p95·커버리지)에 **충족/위반 판정**이 찍힘 · 1-명령 데모 PASS · 검증 테스트 통과.
- **M2 (C3, Should·게이트):** 프로덕션 온프렘에서 `HOST_CLASS=production` p95 측정표 1건 산출 → CMP-147 done. (하드웨어 미제공 시 이 마일스톤은 분리추적으로 잔류, v0.0.6 출시를 막지 않음.)
- **M3 (C2, Could):** 테넌트 A의 조회 세션이 테넌트 B 감사로그를 **못 봄**(격리 테스트 1건) · viewer 역할이 정책 **변경 거부**·조회 허용(RBAC 테스트 1건).
- **출시(release):** C1(+권장/여유 트랙) 완료 후 `VERSION`=0.0.6 · CHANGELOG `[0.0.6]` 항목 · `git tag v0.0.6` — **사람 릴리스 오너**가 수행(CMP-60/137).
