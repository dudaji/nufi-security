# CMP-137 — GOVERNANCE.md 단서 초안: 보드 건별 승인 하 AI 릴리스 메커닉 실행

**트랙:** 거버넌스 설계/명세 (AI 오너 허용 — 설계 트랙). **오너:** CPO(NuFi) 초안 설계. **승인:** 보드. **머지:** 보드 승인 후(규칙 1 — 설계 AI 는 직접 편집 금지, 사람/Engineer 가 머지).

---

## 1. 문제 (문구-운영 긴장)

- **현행 문구(CMP-60 / GOVERNANCE.md 규칙 3):** 릴리스 권한 행위(VERSION 컷·CHANGELOG 확정·`git tag`·push)는 **구현 트랙**으로 분류되어 기본 오너는 **사람 릴리스 오너**(또는 보드 승인 시 Engineer)이며, CEO·CPO·CMO 등 **설계 AI 에이전트는 비배정**이다.
- **CMP-135 운영 실제:** v0.0.2 릴리스 태그 컷에서 보드(@local-board)가 *"CEO가 tag달고 push"* 라고 **명시 건별 지시** → CEO(AI)가 릴리스 메커닉(VERSION/CHANGELOG/tag/push)을 실행함. 근거 코멘트 `76cc5ac9`.
- **판정:** 이번 실행은 **보드(사람·최상위 권한)의 명시 건별 승인** 하였으므로 거버넌스 **위반은 아니다**. 그러나 문구를 글자 그대로 읽으면 "AI 가 릴리스 푸시를 하면 안 된다"로 읽혀 **문구와 운영 사이 긴장**이 남는다. 본 이슈는 그 긴장을 1줄 단서로 해소한다.

## 2. 설계 원칙 (적용 렌즈)

- **리스크 가시성:** 문구-운영 불일치는 leadership 에 보이지 않는 거버넌스 리스크다. 명문화하여 향후 릴리스에서 재해석·논쟁이 없게 한다.
- **권한 최소화:** 예외는 **기계적 집행(이미 확정된 릴리스 내용)** 에만 적용. 제품 코드 작성(규칙 1)으로 번지지 않게 범위를 좁힌다.
- **승인 분리(전이 방지):** 릴리스 푸시 권한은 **빌드 블랭킷 승인**(CMP-96 보안 상시 / CMP-132 v0.0.3 개발 블랭킷)과 **별개**여야 한다. 블랭킷이 릴리스 푸시 권한으로 자동 전이되면 게이트가 무력화된다.
- **마일스톤 위생:** 승인은 **릴리스 단위**로 부여·기록 — 한 릴리스 승인이 다음 릴리스로 전이되지 않는 binary 경계.

>
> **상태:** (보드 승인 시) 시행
> **출처:** 보드 지시 — CMP-135 선례(@local-board *"CEO가 tag달고 push"*, 근거 코멘트 `76cc5ac9`). 본 단서로 CMP-60 문구-운영 긴장 해소.
>
> - **원칙(불변):** 릴리스 권한 행위(VERSION 컷·CHANGELOG 확정·`git tag`·push)는 구현 트랙으로 분류되어 **기본은 AI 에이전트 비배정**이며, 기본 오너는 **사람 릴리스 오너**다(CMP-60).
> - **건별 예외:** 단, **보드(사람·최상위 권한)의 명시적 건별 승인**이 있는 경우, 보드가 **지정한 에이전트**가 해당 릴리스의 메커닉을 실행할 수 있다.
> - **범위 한정:** 본 예외는 **이미 확정된 릴리스 내용의 기계적 집행**(버전 문자열 bump · CHANGELOG 확정 편집 · 태그 · 푸시)에 한정된다. 제품 코드 작성·구현(규칙 1)에는 적용되지 않으며, 릴리스 *내용*(무엇을 릴리스하는가)의 결정 권한을 부여하지 않는다.
> - **블랭킷과 분리:** 본 승인은 **빌드 블랭킷 승인**(CMP-96 보안 상시 승인, CMP-132 v0.0.3 개발 블랭킷 등)과 **별개**다. 빌드 승인이 릴리스 푸시 권한으로 전이되지 않으며, 그 역도 성립하지 않는다.
> - **릴리스 단위 기록:** 승인은 **릴리스 단위로 부여·기록**된다. 한 릴리스의 승인은 다음 릴리스로 전이되지 않는다. 집행 시 릴리스 이슈에 (승인 출처 + 실행 에이전트 + 커밋/태그 해시)를 남긴다.
>
> > 적용: 본 단서는 CMP-60 이슈 게이트 v1(규칙 3)의 보충이며 GOVERNANCE.md(단일 출처)에 추가된다. 확정·머지는 보드 승인 후.
