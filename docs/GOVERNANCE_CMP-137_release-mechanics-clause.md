# CMP-137 — GOVERNANCE.md 개정 초안: CEO=이사회 권한 동격 (자율성 범위 설계)

**트랙:** 거버넌스 설계/명세 (AI 오너 허용 — 설계 트랙). **오너:** CPO(NuFi) 초안. **승인:** 보드. **머지:** 보드 승인 후(규칙 1 — 설계 AI 직접 편집 금지, 사람/Engineer 머지).

> **개정 이력:**
> - rev1 — 보드 건별 승인 하 *릴리스 메커닉* 한정 예외.
> - rev2 — 보드 지시(`f6ef4851`)로 CEO=이사회 권한 동격 격상.
> - **rev3 (현행) — 보드 질의(`ef551735`): "이렇게 되면 이사회에게 아무것도 안 물어보고 계속 작업? 알아서 v0.0.4 기획·작업?" → 자율성 *범위*를 명시 설계.** 권한 동격이 곧 무제한 자율은 아니며, **예약사항(reserved matters)** 으로 보드 통제를 보존하는 모델을 권장안으로 추가.

---

## 1. 보드 질의에 대한 정직한 답

**질문:** CEO=이사회 권한 동격이면, 이사회에 아무것도 안 묻고 계속 작업하나? 알아서 v0.0.4 를 기획·착수하나?

**답 (rev2 문구 그대로라면):** **그렇다.** "이사회 승인 필요" 지점을 CEO 승인으로 충족할 수 있으므로, CEO 가 v0.0.4 를 *기획·자기승인·Engineer 착수* 하는 루프를 보드에 묻지 않고 돌릴 수 있다. (코드는 여전히 Engineer 가 작성하지만, *무엇을 할지 결정·승인*하는 권한이 CEO 에 동격으로 생긴다.)

→ 보드의 우려는 타당하다. **권한 동격(authority)** 과 **무제한 자율(unbounded autonomy)** 은 다르며, 후자는 보드가 의도한 바보다 클 수 있다. 그래서 **자율성의 범위를 명시 설계**하여 보드가 원하는 통제 수준을 고르게 한다.

## 2. 자율성 범위 — 4개 옵션 (보드 선택)

| | 자율 범위 | v0.0.4 예시 | 보드 관여 |
|---|---|---|---|
| **A. 완전 자율** (rev2 그대로) | 기획·승인·착수 전부 CEO 자기승인 | CEO 가 알아서 v0.0.4 기획·착수 | 사후 감사 로그 + 언제든 개입/롤백 |
| **B. 범위 내 자율 + 예약사항** ⭐권장 | *승인된 로드맵 안*에서 자율 | v0.0.4 = **새 버전 라인 → 예약사항 → 보드 승인 필요** | 예약사항만 사전 승인; 그 외 자율 |
| **C. 통지 후 진행** | 자율 승인하되 주요 결정은 사전 통지·침묵시 진행 | v0.0.4 착수 전 보드 통지, 이의 없으면 진행 | 사전 가시성(차단 아님) |
| **D. 현상 유지** | rev2 보류 | v0.0.4 도 건별 보드 승인 | 종전 건별 게이트 |

### 권장: B (범위 내 자율 + 예약사항)

보드가 요청한 "개발 병목 제거"(승인된 작업은 CEO 가 굴림)와 보드 우려("AI 가 알아서 제품 방향을 정하지는 말 것")를 **동시에** 충족한다.

**예약사항(reserved matters) — 사람 보드 승인 유지:**
1. **새 버전 라인/메이저 기획 착수** (예: v0.0.4 시작, 새 제품 트랙) — *제품 방향 결정.*
2. **고객·외부 공개 행위** — 퍼블릭 릴리스 배포·발표·외부 커뮤니케이션·파트너 약정.
3. **거버넌스 변경** — GOVERNANCE.md 개정·철회(자기증식 방지).
4. **에이전트 채용·해임·역할 변경.**
5. **임계 초과 지출/예산 약정.**

**예약사항 외 (CEO 자율):** 이미 승인된 로드맵(예: 현재 v0.0.3) 내 작업의 기획 상세·우선순위·Engineer 착수 승인·릴리스 *메커닉* 실행 등은 CEO 가 보드에 묻지 않고 진행. 모든 자기승인은 이슈 단위 감사 기록 + 보드 상시 개입·롤백권 보존.

## 3. 제안 개정 (규칙 4 — 권장안 B 반영)

> **규칙 4 — CEO 권한 = 이사회 권한 동격 (범위 내 자율 + 예약사항)**
> **출처:** 보드 지시 `f6ef4851` + 자율 범위 질의 `ef551735` (2026-06-27).
>
> 1. **권한 동격.** CEO(AI)는 이사회(보드)와 동일한 권한을 갖는다. "이사회 승인 필요" 지점에서 **CEO 승인 = 보드 승인 효력**(보드도 동격 유지).
> 2. **범위 내 자율.** **이미 보드가 승인한 로드맵·릴리스 라인 안**의 작업은 CEO 가 보드에 건별로 묻지 않고 승인·진행한다(규칙 3-3 게이트를 CEO 승인으로 충족 → 개발 병목 제거).
> 3. **예약사항(보드 승인 유지).** 다음은 권한 동격에서 **제외**되어 **사람 보드의 명시 승인**을 요한다: ① 새 버전 라인/메이저·새 제품 트랙 기획 착수, ② 고객·외부 공개 행위(퍼블릭 릴리스 배포·발표·외부 약정), ③ 거버넌스(GOVERNANCE.md) 개정·철회, ④ 에이전트 채용·해임·역할 변경, ⑤ 임계 초과 지출.
> 4. **릴리스 메커닉.** *이미 승인된* 릴리스 라인의 메커닉(VERSION·CHANGELOG·tag·push) 실행·실행자 지정은 CEO 가 부여, 지정 에이전트 실행, 릴리스 단위 기록. (단 *새 버전 라인 착수* 결정 자체는 ① 예약사항.)
> 5. **구현 독점 불변.** 권한 동격은 승인·지정·의사결정 권한 — 직접 코드 작성은 단일 Engineer 독점(규칙 1·3) 그대로.
> 6. **감사·backstop.** 모든 CEO 자기승인은 이슈 단위 기록. 사람 보드는 모든 결정의 사후 열람·개입·롤백 및 본 규칙 자체의 최종 개정·철회권을 보유.
