## 0. 보드 피드백에 대한 직접 응답

보드 질문 3건에 먼저 답하고(이게 방향을 바꿉니다), 그 위에 다중 후보 + 로드맵을 그립니다.

**Q1. 하이브리드(private–public) 진단용 CLI/SDK가 있어야 하지 않나?**
→ **예, 강력 추천. 현재 없음.** 리포에 있는 CLI 는 `enforcement/cli.py`(`nufi-egress`)뿐이고, 이건 nftables 집행(render/apply/disable/status)에 한정됩니다. **하이브리드 배선을 진단하는 도구는 0.** 하이브리드의 #1 실패 모드는 *조용한 사고* — 트래픽이 게이트웨이를 우회하거나, 탐지가 잘못 배선됐는데 아무도 모르는 것입니다. 그래서 **`nufi doctor` 진단 CLI** 를 제안합니다: 설정 검증 → private/public 경로 도달성 → outbound 가 실제 게이트웨이를 통과하는지 → 우회 누수 탐지(기존 `capture/flow_tap.py` 연계) → **카나리 PII 페이로드를 E2E로 흘려 차단·감사가 실제로 일어나는지 확인**. "지켜지길 바람"을 "검증됨"으로 바꿉니다. 대부분의 LLM 가드레일은 doctor 를 안 줍니다 → **제품 차별점**.

**Q2. 파이프라인 구성을 사용자가 그냥 하게 둬야 하나?**
→ **아니오(raw 로는). 의견 있는 프리셋/프로파일을 주고, raw 는 탈출구로 유지.** 현재 `config/policy.yaml`·`routing.yaml`·`audit_profiles.yaml` 을 손으로 조립합니다. raw YAML 직접 조립은 (1) 전문성 세금이고 (2) 오류 표면입니다 — 정책 한 줄 오설정 = 조용한 유출. **프리셋**(`strict-kr-pii` / `audit-only` / `pseudonymize-roundtrip`)으로 우리의 보안 기본값을 인코딩하면 안전한 채택이 빨라지고 지원 부담이 줍니다. 파워 유저용 raw 편집은 그대로 둡니다(sane defaults + escape hatch).

**Q3. 서빙 시스템 만드는 사용자 관점의 사용 가이드가 지금 있나?**
→ **없음.** 현재 문서는 `README.md`(제품 개요 + Quick Start) + 내부용 SPEC/IMPL/DEMO/STATUS/ARCHITECTURE 뿐입니다. *"나는 LLM 서빙 시스템을 만든다 — NuFi 를 어떻게 앞단에 끼우고, 어떤 프리셋을 고르고, 잘 동작하는지 어떻게 확인하나"* 를 설명하는 **통합/사용 가이드는 0.** 이건 채택의 직접 장애물이라 후보 D1 에 1급으로 넣었습니다.

→ **결론:** 세 질문이 모두 같은 곳을 가리킵니다 — **현재 바인딩 제약은 "탐지 성능"이 아니라 "채택 마찰(Adoptability)"**. 따라서 v0.0.2 의 1순위 후보를 DX 로 재배치합니다.

## 1. 출발점 — v0.0.1 이 끝낸 것 / 안 끝낸 것

**끝남(엔진 "동작 증명"):** 하이브리드 게이트웨이, 한국어 PII·비밀·기밀 인라인 탐지/차단/가명화, 가역 원복+Vault, 100% 감사+해시체인, nftables 우회차단, 비동기 감사봇. 실측 PII recall 0.946 · INT8 p95 38ms · 하드닝 12/12.

**안 끝남(제품화·채택 공백):** 단일명령 배포 없음 · 운영자 룰 갱신에 재기동 필요(NFR3 미충족) · 감사 가시화 화면 없음 · **진단 CLI/클라이언트 SDK 없음** · **서빙빌더 통합 가이드 없음** · **파이프라인 프리셋 없음(raw YAML 수작업)** · NER KR_PERSON INT8 CI 하한 0.832<0.85 · 부하 시 p95 미측정 · retain_raw/키 운영 가이드 미정.

## 2. v0.0.2 후보 방향 5종 (각각 하나의 응집된 릴리스 트랙)

각 트랙 = 무엇 / 왜 / 대표 산출물 / Effort. 단일 테마 강요가 아니라 **보드가 조합을 고를 수 있게** 5종을 제시합니다. 구현 오너 전부 Engineer.

### D1 — Adoptability & Developer Experience (채택·DX) ★보드 질문이 가리키는 곳
- **무엇:** ① 서빙빌더용 **통합/사용 가이드**(게이트웨이 vs SDK vs LiteLLM 훅 중 택일, 프리셋 고르기, 감사로그 읽기, 하이브리드 private/public 결정 트리) ② **thin client SDK**(Python; OpenAI 호환 `base_url` 심 — 한 줄 통합 + 예제) ③ **`nufi doctor` 진단 CLI**(Q1) ④ **파이프라인 프리셋/프로파일 + init 템플릿**(Q2).
- **왜:** 채택 마찰이 지금의 바인딩 제약. 가이드·SDK·진단·프리셋 없이는 배포(D2)해도 오용되고 실트래픽 PoC(KR a)에 못 간다.
- **Effort:** M–L (가이드 S · SDK S · doctor M · 프리셋 M)

### D2 — Pilot-Ready / Operability (배포·운영)
- **무엇:** 단일명령 배포(Docker Compose + 에어갭 이미지 번들 + Helm 스텁) · 정책 핫리로드+드라이런 · 감사 가시성 대시보드(read-only, Langfuse/Grafana 위) · retain_raw·키 회전 하드닝.
- **왜:** 고객이 온프렘에서 *운영* 가능. NFR3 충족.
- **Effort:** M–L

### D3 — Accuracy & Performance Hardening (정확도·성능 부채)
- **무엇:** NER base 모델 격상(KR_PERSON CI 0.85↑) · 동시성/지속부하 p95 재측정(NFR2 실환경) · 스트리밍 원복 견고화.
- **왜:** v0.0.1 의 정의된 잔여 부채. recall≥0.9·지연 SLA 를 실트래픽에서 방어.
- **Effort:** M

### D4 — Detection Depth / DLP (탐지 깊이)
- **무엇:** 기밀 3계층 IDM 문서지문 · 임베딩 분류기(패러프레이즈 일반화, 라벨 필요) · 기밀 커버리지 확대.
- **왜:** 차별화된 DLP. **단 디자인파트너 수요 의존** — 수요 미확인 선투자 비권장.
- **Effort:** L–XL

### D5 — Coverage Assurance (우회·커버리지 보증) — D1 진단의 상시 런타임판
- **무엇:** "내 트래픽 중 몇 %가 실제로 게이트웨이를 통과했나" 커버리지 리포트 + 우회 경로 지속 모니터링/알림. nftables 집행을 *측정 가능한 보증*으로.
- **왜:** 하이브리드 핵심 실패=조용한 우회. `nufi doctor`(1회 진단)의 상시 감시 버전.
- **Effort:** M

## 3. 큰 로드맵 (Horizon 배치)

세 호라이즌으로 트랙을 배치합니다. **Adopt → Operate → Scale/Differentiate.**

```
            │ Adopt (v0.0.2)        │ Operate (v0.0.3)       │ Scale·Differentiate (v0.1.0)
────────────┼───────────────────────┼────────────────────────┼──────────────────────────────
 DX/채택    │ D1 가이드·SDK·doctor   │ 프리셋 카탈로그 확장     │ 멀티테넌트 콘솔
            │    ·프리셋 (척추)      │                        │
 운영       │ D2 배포패키징(슬라이스)│ D2 대시보드·핫리로드 완성│ 멀티테넌시·RBAC
 보증       │ (doctor=1회 진단)      │ D5 커버리지 보증(상시)  │ D5 SLA 리포팅
 정확도     │ D3 NER격상·부하p95     │ D3 스트리밍 견고화 완료  │ 대규모 하드닝
 탐지깊이   │ —(수요 관망)           │ D4 IDM 문서지문(수요시) │ D4 임베딩 분류기(수요시)
────────────┴───────────────────────┴────────────────────────┴──────────────────────────────
 릴리스 주제 │ "서빙빌더가 반나절에   │ "보안팀이 온프렘에서    │ "다고객·차별화 DLP·
            │  채택·검증"           │  운영·관측·보증"        │  규모 SLA"
```

호라이즌은 순서지 마감일이 아닙니다. D4(탐지깊이)는 **디자인파트너 수요 신호가 뜨면** 어느 호라이즌으로든 당겨집니다.

## 4. v0.0.2 권장 구성 (CPO 추천)

보드 질문이 DX 를 가리키므로 **D1 을 v0.0.2 척추로** 두고, 실배포에 꼭 필요한 D2 슬라이스와 정확도 부채 D3 를 병행합니다.

| 등급 | 포함 | 근거 |
|---|---|---|
| **Must** | **D1** 통합 가이드 + thin SDK + `nufi doctor` + 프리셋 | 채택 마찰 해소 = 실트래픽 PoC(KR a)의 전제. 보드 Q1·Q2·Q3 직접 응답 |
| **Must** | **D2 슬라이스** 단일명령 배포(compose + 에어갭 번들) | 진단·SDK 가 있어도 올릴 단위가 없으면 PoC 불가 |
| **Should** | **D3** NER base 격상 + 부하 p95 재측정 | recall≥0.9·지연 SLA 실환경 방어(v0.0.1 잔여) |
| **Should** | **D2 잔여** 정책 핫리로드 + retain_raw/키 하드닝 | NFR3 + 보안검토 통과 |
| **Could** | **D5** 커버리지 보증(doctor 의 상시판) · D2 감사 대시보드 | 여유/수요 시 |
| **Won't(이번)** | **D4** IDM/임베딩 · 멀티테넌시 · TLS MITM | 수요 미확인·범위·리스크 |

**시퀀스:** M1 *Adopt 코어*(D1 가이드+SDK+doctor+프리셋, D2 배포) → M2 *Hardening*(D3 NER·부하, D2 핫리로드·키) → M3 *Assurance*(D5, 대시보드; 여유 시).

각 마일스톤 binary 완료기준(예시):
- M1: 신규 사용자가 가이드만 보고 SDK 한 줄로 통합 → `nufi doctor` 가 "게이트웨이 통과·PII 차단·감사적재" 3 체크 GREEN → 프리셋 1종 적용. `docker compose up` 1회 기동.
- M2: 재기동 없이 정책 룰 1건 변경 적용(드라이런 diff) · KR_PERSON CI 하한 재측정 기록 · 동시성 p95 1건 리포트.
- M3: 커버리지 리포트 1건("X% 게이트웨이 통과") · 대시보드에서 결정 1건 조회.
