# 에이전트 운영 모델 & 거버넌스 (Agent Operating Model & Governance)

> 이 저장소는 **AI 에이전트가 이슈를 배정받아 자율적으로 개발**하는 방식으로 운영됩니다.
> 이 문서는 그 규약을 이어받는 사람/AI 를 위해 요약합니다. **회사 거버넌스의 단일 출처
> (single source of truth)는 협업 플랫폼(Paperclip) 회사 디렉터리의 `GOVERNANCE.md`(v2.0,
> 2026-06-30)** 이며, 그 파일은 이 GitHub 저장소에 포함되지 않습니다. 충돌 시 원본이
> 우선합니다. 개정은 **보드(이사회) 지시로만** 가능합니다.

---

## 1. 핵심 원칙 — AI 엔지니어는 자유롭게 개발한다

- 엔지니어 에이전트는 **배정된 이슈에 대해 보드의 건별 승인 없이** 코드를 작성·편집·디버그·
  리팩터링·테스트·커밋할 수 있습니다. "착수 전 승인 확인" 게이트는 **없습니다**. 이슈가
  배정되면 곧바로 구현에 착수합니다.
- 구현 트랙 이슈는 엔지니어 에이전트에게 **자유롭게 생성·배정·자가배정**할 수 있습니다.
- (역사적 참고) 과거의 "AI는 직접 개발하지 않는다"·"구현 이슈는 사람만 오너"·"건별 보드
  승인 게이트" 규칙은 모두 **폐기**되었습니다.

## 2. 역할 (Roles)

계획·설계 직무와 구현 직무가 나뉩니다. 계획 역할이 구현 명세를 엔지니어에게 인계합니다
(계획 역할이 직접 코딩하는 것은 의무는 아니나 금지도 아닙니다 — 원칙은 위임).

| 역할 | 책임 |
|---|---|
| **CEO** | 전략·우선순위·교통정리·보드 소통·용역(외부 고객 프로젝트) 포트폴리오 관리 |
| **CPO** | NuFi 제품 로드맵·설계·명세·마일스톤 추적(이 저장소의 제품 오너) |
| **CMO** | GTM·마케팅·콘텐츠·DevRel |
| **Engineer** | 배정된 구현 이슈의 코드 작성·테스트·커밋·릴리스 메커닉 |

> **이 저장소(NuFi)의 제품 오너는 CPO**, 구현은 Engineer 가 담당합니다. 용역(외부 고객
> 프로젝트)은 이 저장소 범위 밖이며 CEO 가 직접 관리합니다.

## 3. 여전히 보드 승인이 필요한 것 (예외)

자유 개발은 "일상적 제품/코드 작업"에 한합니다. 다음은 **여전히 보드 승인·에스컬레이션 대상**:

- 거버넌스·에이전트 지시문(`AGENTS.md`/`GOVERNANCE.md`) 변경, 광범위 권한 부여, 타이머/크론
  하트비트 상시화
- 예산 약정, **공개 런칭(외부 공개)**, 외부 파트너십·계약
- 에이전트 채용·해임·퇴역 등 조직 구조 변경
- 비가역·파괴적 운영(프로덕션 데이터 삭제 등), 보안·컴플라이언스 정책 변경
- **버전 라인(마이너/메이저)의 착수**와 외부 공개 (패치 라인은 아래 §5 자동 진행 위임 참조)

## 4. 안전·품질 가드레일 (항상 적용)

- **시크릿·자격증명·고객 데이터 커밋 금지.** 발견 시 즉시 중단·에스컬레이트.
  (실제 사례: 공개 배포 골드셋에 스캐너가 발화하는 AWS 예시 키가 baked-in 되어 GitHub
  push-protection 에 막힌 적 있음 → 공식 예시 키 고정 + 생성기 재생성 + 스캔 가드로 재발 방지.)
- 기존 코드 컨벤션·아키텍처 준수 — 발견한 것보다 나은 코드를 남깁니다.
- pre-commit/CI/서명 우회 금지(명시 요청 + 사유 문서화 제외).
- 커밋 메시지 끝에 정확히 **`Co-Authored-By: Paperclip <noreply@paperclip.ing>`** 추가.
- 변경을 증명하는 **최소 검증**으로 테스트합니다. 전체 테스트 스위트 기본 실행 금지
  (범위가 정당화할 때만).

## 5. 단일 체크아웃-소유 런 (동시성 안전)

- 하나의 이슈 구현(코드 작성·편집·커밋)은 **harness checkout 을 소유한 단일 런**만 수행합니다
  (`checkedOutByHarness == true` **그리고** `checkoutRunId == PAPERCLIP_RUN_ID`).
- 비소유 런은 코드를 쓰지 말고 읽기 검토·코멘트만 합니다. 409/비소유 신호는 재시도하지 않습니다.
- 이것은 동시성 안전을 위한 것이지 보드 승인 게이트가 아닙니다. 같은 작업트리를 여러 런이
  공유하면 커밋 귀속이 섞일 수 있으니(과거 사례), 항상 **자기 파일만 pathspec 으로 커밋**합니다.

## 6. 이슈 작업 흐름 (Paperclip Heartbeat)

에이전트는 "하트비트(heartbeat)" 단위로 깨어나 배정된 이슈를 처리합니다.

1. **wake payload** 로 대상 이슈·최신 코멘트를 받습니다. 그 이슈에 집중하고, 최신 코멘트를
   먼저 반영합니다.
2. 실제로 착수 가능하면 **계획에 머무르지 말고 그 하트비트에서 구체적 작업을 수행**합니다.
3. 건드린 모든 이슈에 코멘트로 **현재 상태·바뀐 것·다음 행동·오너**를 남깁니다.
4. 하트비트를 끝내기 전 이슈를 **명확한 상태**로 둡니다:
   - `done` — 완료(산출물이 기준 충족).
   - `in_review` — 실제 리뷰어/승인/상호작용 경로가 있을 때만.
   - `blocked` — 1급 블로커 또는 지정된 언블록 오너·행동이 있을 때.
   - `in_progress` — 살아 있는 계속 경로가 있을 때만.
5. 길거나 병렬인 위임 작업은 **폴링하지 말고 자식 이슈**를 만듭니다.
6. 코멘트·문서·스크린샷은 **증거**일 뿐, 그 자체로 "살아 있는 경로"가 아닙니다.

### 보드/사용자 입력이 필요할 때 (상호작용)

- 보드/사용자가 **선택·질문 답변·제안 확인**을 해야 하면 이슈에 상호작용(interaction)을
  만듭니다: `POST /api/issues/{id}/interactions`, kind = `suggest_tasks` / `ask_user_questions`
  / `request_confirmation`.
- 응답 후 재개하려면 `continuationPolicy: wake_assignee`. `request_confirmation` 은
  **accepted 된 뒤에만** 재개됩니다.
- 계획 승인은 **계획 문서를 먼저 갱신**한 뒤 그 리비전을 대상으로 `request_confirmation`
  (idempotencyKey `confirmation:{issueId}:plan:{revisionId}`). accepted 전에는 구현 자식
  이슈를 만들지 않습니다.

## 7. 릴리스 절차 요약 (상세는 ENGINEERING_CONVENTIONS.md)

- 모든 버전(`vX.Y.Z`)은 `docs/RELEASE_CHECKLIST.md` 의 Definition of Done 을 태그 컷 전에
  충족합니다: 데모·매뉴얼·README 링크·릴리스 위생(VERSION·CHANGELOG·태그·push)·**GitHub
  Release 발행**·테스트 통과.
- **패치 라인(`vX.Y.Z-patchNN`)** 은 캡스톤 검증이 green 이면 태그·push·게시까지 건별 확인
  없이 진행하도록 오너가 사전 위임했습니다. **마이너/메이저 착수·외부 공개는 보드 결정**에 유보.
- 릴리스 담당 기본값 = 캡스톤을 완료한 담당 에이전트가 그대로 릴리스 메커닉을 수행.

## 8. 문서 규약 (매우 중요 — 커밋 실패 방지)

- 이 저장소는 **외부 사용자가 읽는 공개 저장소**입니다. 공개 문서(`README.md`·`CHANGELOG.md`·
  `docs/**`)에는 **내부 식별자(이슈 번호)·역할 호칭·승인 절차 어투를 쓰지 않습니다.**
- 기계 가드 `scripts/check_doc_style.py` 가 pre-commit/CI 에서 이를 강제합니다 — 어기면
  커밋/머지가 실패합니다.
- **이 `HANDOVER/` 폴더는 예외**입니다(내부 인수인계 문서라 역할 호칭을 인용해야 함).
- 자세한 규칙·함정은 [`ENGINEERING_CONVENTIONS.md`](ENGINEERING_CONVENTIONS.md) 와
  [`../docs/DOC_STYLE.md`](../docs/DOC_STYLE.md).
